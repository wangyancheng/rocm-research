# 关于 HSA 平台系统架构规范

本文档从硬件角度识别了支持异构系统架构 (HSA) 编程模型以及 HSA 应用和系统软件基础设施所需的系统架构需求。

它定义了一组功能与特性，供 HSA 硬件产品交付物满足最低规定要求，以符合有效 HSA 产品的资格。

在必要时，文档会说明可能的设计实现以阐明预期操作。除非另有规定，这些实现无意暗示特定的硬件或软件设计。

## 受众

本文档面向对在平台设计中支持 HSA 基础设施（硬件和软件）感兴趣的系统及组件架构师。

## 术语

术语定义请参见附录 B 术语表（第 72 页）。

本规范使用 C 语言家族的术语和语法。例如，诸如 `uint64_t` 的类型名在 C99 和 C++ 规范中定义。

## HSA 信息来源

*   HSA 程序员参考手册 版本 1.1
*   HSA 运行时程序员参考手册 版本 1.1
*   HSA 平台系统架构规范 版本 1.1

# 第 1 章. 系统架构需求：概述

## 1.1 什么是 HSA？

异构系统架构 (HSA) 旨在高效支持各种数据并行和任务并行编程模型。单个 HSA 系统可以支持基于主机 CPU 和内核代理的多种指令集。

HSA 支持两种机器模型：大型模型（64 位地址空间）和小型模型（32 位地址空间）。

一个 HSA 兼容系统将满足队列模型、内存模型、服务质量以及并行处理指令集的要求。它还将满足使用标准化接口、进程、通信协议和内存模型为计算平台启用异构编程模型的要求。

## 1.2 关键词

本文档使用以下定义的关键词来指定不同级别的 HSA 系统需求：

- **"必须" (Must)**：此词，或术语 **"要求" (required)** 或 **"应" (shall)**，意味着该定义是规范的绝对要求。
- **"不得" (Must not)**：此短语，或短语 **"不应" (shall not)**，意味着该定义是规范的绝对禁止。
- **"应该" (Should)**：此词，或形容词 **"推荐" (recommended)**，意味着在特定情况下可能存在忽略某个特定项的有效理由，但在选择不同做法之前，必须理解并仔细权衡全部影响。
- **"不应该" (Should not)**：此短语，或短语 **"不推荐" (not recommended)**，意味着在特定情况下，当特定行为可接受甚至有用时，可能存在有效理由，但在实施任何带有此标签的行为之前，应理解全部影响并仔细权衡情况。
- **"可以" (May)**：此词，或形容词 **"可选" (optional)**，意味着某个项是真正可选的。一个供应商可能因为特定市场需要它，或者因为供应商认为它能增强产品而选择包含该项，而另一个供应商可能省略同一项。未包含特定选项的实现**必须**准备好与包含该选项的另一实现进行互操作，尽管可能功能有所降低。同样，包含特定选项的实现**必须**准备好与未包含该选项（当然，除了该选项提供的功能之外）的另一实现进行互操作。

这些定义完全按照 Internet 工程任务组 RFC 2119, BCP 14,《用于 RFC 中以指示需求级别的关键词》中的描述。

## 1.3 最小 vs. 完整 HSA 软件产品

本文档也为 HSA 产品更完整、增强和有竞争力提供了指导：

- 最小 HSA 软件产品由关键词 **"应" (shall)**、**"必须" (must)** 和 **"要求" (required)** 突出显示的强制性需求定义。
- 完整 HSA 软件产品功能集由关键词 **"应该" (should)** 和 **"推荐" (recommended)** 定义。

除非另有说明，本文档引用的 HSA 软件之外的功能（例如软件组件或 API）指的是 HSA 软件交付时当前的版本。

**注意：** 如果存在隐含的矛盾，更高级别的需求规范应取代较低级别（内核代理）的需求。

## 1.4 HSA 编程模型

HSA 编程模型通过异构系统组件（例如内核代理和其他代理、接口连接结构、内存等）中存在的一组关键硬件和系统特性来启用。这些特性在 HSA 兼容系统上的存在简化了软件栈需要处理的排列组合数量。因此，HSA 编程模型比基于更传统系统设计的异构系统编程模型简单得多。

## 1.5 HSA 兼容系统需求

一个能够通过所有 HSA 一致性测试的系统是以下组件的组合：

- 一个或多个执行 HSA 运行时的主机 CPU 代理（参见 HSA 运行时程序员参考手册 版本 1.1），
- 一个或多个能够执行 HSAIL 程序的内核代理（参见 HSA 程序员参考手册 版本 1.1），
- 以及零个或多个参与 HSA 内存模型的其他代理（参见 HSA 平台系统架构规范 版本 1.1）

同时还需遵守本文档中定义的以下需求：

- **共享虚拟内存**，包括遵守 HSA 内存模型。参见 2.1 需求：共享虚拟内存（第 11 页）。
- **缓存一致性域**，包括主机 CPU、内核代理和其他代理以及互连 I/O 总线结构。参见 2.2 需求：缓存一致性域（第 13 页）。
- **内存扁平寻址**。参见 2.3 需求：扁平寻址（第 13 页）。
- **一致的系统端序**。参见 2.4 需求：端序（第 13 页）。
- 所有启用 HSA 的系统组件之间基于内存的**信令和同步原语**，包括对平台原子操作的支持。参见 2.5 需求：信令和同步（第 14 页）。
- **原子内存操作**，参见 2.6 需求：原子内存操作（第 16 页）。
- **HSA 系统时间戳**，在整个 HSA 系统中提供统一的时间视图。参见 2.7 需求：HSA 系统时间戳（第 17 页）。
- **用户模式队列**，具有低延迟的应用级派发到硬件的能力。参见 2.8 需求：用户模式队列（第 17 页）。
- 使用**架构化队列语言 (AQL)**，通过允许应用程序将任务入队到内核代理和其他代理来减少启动延迟。参见 2.9 需求：架构化队列语言 (AQL)（第 24 页）。
- **代理调度**，参见 2.10 需求：代理调度（第 30 页）。
- 如 2.11 需求：内核派发前进保证（第 31 页）中定义的**内核派发前进保证**。
- **内核代理错误报告机制**，其详细程度与主机 CPU 提供的类似，包括遵守 2.12 需求：IEEE754-2008 浮点异常（第 32 页）中规定的策略。
- **内核代理调试基础设施**，满足规定的功能支持级别。参见 2.13 需求：内核代理硬件调试基础设施（第 33 页）。
- 通过 ACPI 或等效的架构化固件基础设施提供的系统固件表进行的**架构化内核代理发现**。这允许系统软件和应用程序软件发现和利用平台拓扑，而无需依赖于系统特定的总线结构和主机 CPU 基础设施，只要它们支持其他 HSA 相关功能即可。参见 2.14 需求：HSA 平台拓扑发现（第 34 页）。
- 可选地，HSA 平台支持**图像操作**。参见 2.15 需求：图像（第 40 页）。
- 可选地，HSA 平台支持对 HSA 软件进行**性能分析**。参见 2.16 需求：性能分析（第 43 页）。

注意，实现这些需求的方法多种多样。

# 第 2 章. 系统架构需求：细节

## 2.1 需求：共享虚拟内存

一个兼容的 HSA 系统**应**允许代理通过公共的 HSA 统一虚拟地址空间访问共享系统内存。对于 64 位 HSA 系统实现，所有代理必须支持的最小虚拟地址宽度为 48 位；对于 32 位 HSA 系统实现，则为 32 位。完整的可寻址虚拟地址空间**应**可用于指令和数据。

指针在 32 位 HSA 系统中存储在 32 位字中，在 64 位 HSA 系统中存储在 64 位字中。

系统软件可以保留虚拟地址空间的范围供代理或系统内部使用，例如私有段和组段。特定代理对这些范围内的位置的访问遵循实现特定的系统软件策略，并且不受本规范进一步定义的共享虚拟内存需求和访问属性的约束。保留的范围必须可由系统软件发现或配置。系统软件预期不会为 HSA 应用程序访问在这些不可共享范围内分配页。

基础配置文件中放宽了对共享虚拟内存的要求，仅适用于通过 HSA 运行时内存分配器分配的缓冲区。基础配置文件内核代理**必须**支持可分配的所有全局段内存的缓冲区内的细粒度共享。使用基础配置文件内核代理的应用程序可以选择不分配所有具有细粒度共享的全局段缓冲区。

类似地，全局段内存也可以分配用于内核参数，除了系统中全局段内存的总量外，对此类内存的总量没有任何限制。

每个代理**应**通过由系统软件管理的页表处理共享内存虚拟地址转换。系统软件负责根据其策略设置、维护和失效这些页表。代理**必须**遵守由系统软件建立的共享内存页属性。所观察到的共享内存页属性在所有代理间**应**保持一致。

共享内存虚拟地址转换**应**：

- 为所有代理一致地解释共享内存属性，并确保内存保护机制不能被规避。特别是：
    - 所有代理**应**支持相同的页大小。
    - 读写权限平等地适用于所有代理。
    - 执行限制不要求应用于内核代理。
- 代理**必须**支持最低特权级别的共享虚拟内存。
- 代理不要求为主机 CPU 操作系统或 hypervisor 可能使用的更高级特权级别支持共享虚拟内存。
- 代理对共享虚拟内存的执行访问**必须**仅使用最低特权级别。
- 如果在 HSA 系统中实现，页状态位更新（例如，"访问" 和 "脏"）**必须**跨所有代理跟踪，以确保正确的分页行为。
- 对于主内存类型，所有代理（包括主机 CPU）**必须**以相同方式解释可缓存性和数据一致性属性（不包括只读图像数据的属性）。
- 对于主内存类型，代理**应**以通用方式解释内存类型，并与主机 CPU 相同，但有以下例外：
    - 端点排序属性。
    - 观察排序属性。
    - 多副本原子性属性。
- 对于主内存类型之外的任何内存类型，代理**应**要么：
    A. 在使用该内存类型时生成内存故障，或者
    B. 以通用方式解释内存类型，并与主机 CPU 相同，但有以下例外：
        - 端点排序属性。
        - 观察排序属性。
        - 多副本原子性属性。
        - 可缓存性和数据一致性属性。
- 对于所有内存类型，要求所有代理和主机 CPU 对推测许可属性有相同的解释。
- 提供一种机制，在发生转换故障时通知系统软件。此通知**应**包括虚拟地址和用于识别发出转换请求的代理的设备标签。
- 提供一种机制来处理可恢复的转换故障（例如，页不存在）。系统软件**应**能够启动对触发转换故障的地址转换请求的重试。
- 在协议中提供进程地址空间 ID (PASID) 的概念，以服务系统中独立的、按进程的虚拟地址空间。对于支持硬件虚拟化的系统，PASID 实现**应**允许将 PASID 位分离用于虚拟化（例如，作为 "PartitionID"）和用于 HSA 内存管理单元 (HSA MMU) 使用的 PASID 位。用于 HSA MMU 功能的 PASID 位数**应**至少为 8 位。如果 HSA 系统面向更高级的系统软件（同时运行许多进程），建议为所有代理支持更高的最小 PASID 位数（16 位）。

## 2.2 需求：缓存一致性域

所有代理对全局内存的数据访问**应**是一致的，无需显式缓存维护。这仅适用于转换系统的主内存类型的全局内存位置，不适用于图像访问（有关图像的详细信息，请参见 2.15 需求：图像（第 40 页））。HSA 应用程序可以限制数据项的一致性范围作为性能优化。详细信息请参见第 3 章 HSA 内存一致性模型（第 45 页）。

规范不要求代理对具有转换系统支持的主内存类型之外的任何内存类型的任何内存位置的数据内存访问是一致的。

规范不要求代理对任何内存类型的指令内存访问是一致的。

规范不要求代理对任何未指定相同内存属性的内存位置具有一致的视图。

HSA 共享虚拟内存别名指向同一物理地址时，一致性和排序不作保证。

### 2.2.1 只读图像数据

只读图像数据要求在 HSA 内核执行期间保持静态。其影响包括（但不限于）不允许使用同一数据的读/写图像视图来修改只读图像视图的数据，或者从相同内核、主机 CPU 或并行运行的另一个内核直接访问数据数组。

## 2.3 需求：扁平寻址

内核代理**必须**为其可以访问的所有内存支持扁平虚拟地址空间。对扁平地址的 HSAIL 操作等同于指定相应段的相同 HSAIL 操作。为 HSAIL 内存操作指定段可能会提高性能。

使用扁平地址的同步内存操作适用于该扁平地址指定的段。

使用扁平地址的同步内存操作的有效内存作用域是该操作指定的作用域与该扁平地址段支持的最宽作用域中的较小者。

扁平地址空间中段的基地址**必须**对齐到 4kB。

全局段中某个位置的地址值与其在扁平地址空间中的等效位置相同。

扁平地址空间中某个位置的地址值与其在主机虚拟地址空间中的等效位置相同。

NULL 值对于全局段和扁平地址空间是相同的，并且与主机平台的 NULL 值相同。对于其他段，NULL 可能被转换为私有值。

## 2.4 需求：端序

HSA 系统可以实现为小端序或大端序，假设系统中协作的代理只使用一种端序。本文档（特别是在定义数据结构的表格中）使用以下关于端序的约定：

- 位分配到字节：位 0..7 存储在字节 0 中，位 8..15 存储在字节 1 中，依此类推。
- 多位字段中的位值：本文档未定义在多位字段中为每个位分配数值的方法，但系统中的所有代理**必须**具有相同的定义。

## 2.5 需求：信令和同步

兼容 HSA 的平台**应**提供能力，使 HSA 软件能够定义和使用可从所有代理以非歧视性方式访问的信令和同步原语。HSA 信令原语用于操作信号值，该值使用不透明的 64 位信号句柄引用。除了信号值之外，信号句柄还可能关联了实现定义的数据。在本文件中，信号值和实现数据统称为 HSA 信号或简称为信号。

信令机制具有以下属性：

- HSA 信号值**必须**仅由内核代理使用特定的 HSAIL 机制，以及由主机 CPU 使用 HSA 运行时机制进行操作。其他代理可以使用与 HSA 运行时和 HSAIL 操作信号机制兼容的实现特定机制。HSAIL 和 HSA 运行时所需的机制如下：
    - **分配 HSA 信号**：
        - HSA 系统**应**支持通过 HSA 运行时使用专用的运行时 API 来分配信号。应用程序可以通过运行时代理派发来公开此功能。
        - HSA 信号的分配**必须**包括信号值的初始化。
        - 应用程序**可以**选择性地指定需要通过信号上的发送或原子读-改-写操作来通知的代理集。如果应用程序未指定任何要通知的代理集，则信号上的发送或原子读-改-写操作要求通知所有代理。
    - **销毁 HSA 信号**：
        - HSA 系统**应**支持通过 HSA 运行时使用专用的运行时 API 来释放信号。应用程序可以通过运行时代理派发来公开此功能。
    - **读取当前 HSA 信号值**：
        - 信号读取机制**必须**支持宽松 (relaxed) 和获取 (acquire) 内存排序语义。读取 HSA 信号的值会返回该 HSA 信号的瞬时值，如该执行单元所见（有关内存一致性的信息，请参见 2.15 需求：图像（第 40 页））。不保证代理会看到信号的所有顺序值。
    - **等待 HSA 信号满足指定条件**：
        - 所有信号等待操作都指定一个条件。支持的条件是：等于、不等于、小于、大于或等于。
        - 在评估等待条件时，信号值被视为有符号整数。
        - 当通过发送或原子读-改-写操作修改信号时，将安排对该信号上的所有等待操作进行条件检查。条件的检查和等待操作执行单元的潜在恢复也可能被其他因素（例如操作系统进程调度）阻塞。
        - 等待操作不一定会看到信号的所有顺序值。应用程序**必须**确保信号的使用方式使得在依赖的执行单元唤醒之前，等待唤醒条件不会失效，否则唤醒可能不会发生。
        - 信号等待操作可能会零星返回。不能保证返回时信号值满足等待条件。调用者**必须**确认信号值是否已满足等待条件。
        - 等待操作有一个最大持续时间，之后会返回，该持续时间可以通过 HSA 运行时基于 HSA 系统时间戳频率以计数形式发现。等待操作执行单元的恢复也可能被其他因素（例如操作系统进程调度）阻塞。
        - 信号等待机制**必须**支持宽松 (relaxed) 和获取 (acquire) 内存排序语义。
        - **必须**允许多个代理同时等待同一个 HSA 信号。
    - **等待 HSA 信号满足指定条件，并请求最大等待持续时间**：
        - 对普通信号条件等待操作的要求也适用于此操作。请求的最大等待持续时间是一个提示。即使条件未满足，该操作的阻塞时间也可能更短或更长。最大等待持续时间请求使用与 HSA 系统时间戳相同的单位指定。
    - **发送 HSA 信号值**：
        - 发送信号值**应该**唤醒任何在该信号上等待且等待条件得到满足的代理。唤醒是否仅针对等待条件满足的代理发出，是实现定义的。
        - 信号发送机制**必须**支持宽松 (relaxed) 和释放 (release) 内存排序语义。
    - **原子读-改-写 HSA 信号值**：
        - 除了 `wrapinc`、`wrapdec`、`min` 和 `max` 之外，HSAIL 中为通用数据定义的同一套原子操作**应该**在 HSAIL 和 HSA 运行时中都支持用于信号。
        - 对信号值执行原子读-改-写更新**应该**唤醒任何在该信号上等待且等待条件得到满足的代理。
        - 唤醒是否仅针对等待条件满足的代理发出，是实现定义的。
        - 信号原子更新机制**必须**支持宽松 (relaxed)、释放 (release)、获取 (acquire) 和获取-释放 (acquire-release) 内存排序语义。
        - 在由运行时为与内核代理关联的用户模式队列创建的队列门铃信号上，不支持原子读-改-写操作。

HSA 信号**应该**仅由分配它的同一用户进程地址空间内的代理操作。HSA 信令机制不支持进程间通信。

对 HSA 信号的数量没有架构上的限制，除非受本文档中概述的其他需求隐式限制。

HSA 信号句柄始终是一个 64 位宽的量，具有不透明格式：

- HSA 信号句柄值为零**必须**解释为没有信号。
- HSA 信号值的宽度取决于支持的机器模型：
    - 在小型机器模型中使用时，HSA 信号**必须**具有 32 位信号值。
    - 在大型机器模型中使用时，HSA 信号**必须**具有 64 位信号值。

信号通常用于代理之间的通知。因此，信号操作与内存模型的交互方式，就好像信号值驻留在全局段内存中，是自然对齐的，并且是使用原子内存操作在系统作用域内以信号操作指定的内存排序进行访问的。有关内存一致性的更多信息，请参见第 3 章 HSA 内存一致性模型（第 45 页）。

## 2.6 需求：原子内存操作

HSA 要求内核代理支持以下标准原子内存操作（其他代理仅需支持其在系统中所扮演角色所需的这些操作的子集）：

- 从内存加载
- 存储到内存
- 从内存获取，应用带有一个额外操作数的逻辑操作，并存储回去。所需的逻辑操作是按位与、按位或、按位异或。
- 从内存获取，应用带有一个额外操作数的整数算术操作，并存储回去。所需的整数算术操作是加、减、递增、递减、最小、最大。
- 与操作数交换内存位置。
- 比较并交换，即加载内存位置，与第一个操作数比较，如果相等，则将第二个操作数存储回内存位置。

在小型机器模型中，**应该**支持 32 位标准原子内存操作；大型机器模型要求支持 32 位和 64 位标准原子内存操作。

## 2.7 需求：HSA 系统时间戳

HSA 系统提供了一种低开销的机制来确定时间的流逝。要求提供一个系统时间戳，可以从 HSAIL 或通过 HSA 运行时读取。还可以通过 HSA 运行时确定系统时间戳的频率。

HSA 系统要求时间戳具有以下属性：

- 时间戳值是 64 位整数。
- 时间戳值**不应该**发生回绕。
- 无论系统可能进入何种低功耗状态或工作频率发生任何变化，时间戳值**应该**以恒定速率增加，对 HSA 应用程序可见。
- 可观察到的恒定时间戳值增加速率在 1-400MHz 范围内，并可通过 HSA 运行时查询。
- 在整个 HSA 系统中提供统一的时间视图，即：
    - 对于单个读取者，系统时间戳的值**必须**永远不会出现随时间减少的情况。
    - HSA 系统时间戳功能**必须**生成单调递增的时间戳值，使得从每个代理的角度看，观察到先生成的时间戳 A 的值小于或等于后生成的时间戳 B 的值。

不要求时间戳不回绕可以通过多种方式解决。例如，如果时间戳值不与现实世界时间同步，并且在每次系统启动时重置，那么需要设计的回绕周期可以是最大预期系统正常运行时间，而不是整个生命周期。其他实现可能希望为 HSA 系统时间戳重用现有的现实世界时间戳。在这种情况下，不回绕的要求将意味着不同的硬件要求。

系统时间戳功能可以通过多种方式实现。可能的实现示例如下，但只要满足要求，许多其他方式也是可能的：

- 系统中的单个时间戳计数器外设，在被代理读取时生成时间戳。
- 以相同速率递增并具有同步启动的分布式本地计数器。

实现可以改变时间戳每次递增的量，例如在进入或退出低功耗状态时，只要满足上述要求即可。

## 2.8 需求：用户模式队列

兼容 HSA 的平台**应**支持分配多个 "用户级命令队列"。

对于 HSA 软件而言，用户级命令队列**应**被描述为运行时分配的、用户级、可访问的虚拟内存，具有特定大小，包含由架构化队列语言定义的数据包（参见 2.9 需求：架构化队列语言 (AQL)（第 24 页））。HSA 软件**应该**接收基于内存的结构来配置硬件队列，以便高效地对代理的硬件队列进行软件管理。

此队列内存**应**由数据包处理器作为环形缓冲区处理，并使用单独的内存位置来定义该队列的写入和读取状态信息。数据包处理器在逻辑上是与内核代理分离的代理，例如，在分配信号时可能具有作为监听者的独立标识。

**用户模式队列应定义如下：**

**表 2-1 用户模式队列结构**

| 位域 | 字段名 | 描述 |
| :--- | :--- | :--- |
| 31:0 | type | 队列类型，用于未来扩展。旨在用于动态队列协议确定。队列类型在表 2-2（下一页）中定义。 |
| 63:32 | features | 位域，用于指示队列支持的具体特性。HSA 应用程序应忽略任何未知的设置位。<br>特性与类型的不同之处在于，特定类型的队列可能具有某个特性（行为上的特定限制、支持额外的数据包类型），但仍然与该基本队列类型兼容。<br>队列特性在 2.8.2 队列特性（下一页）中定义。 |

| 191:128 | doorbell_signal | HSA 信令对象句柄，当新的 AQL 数据包入队时，必须用最新的 `write_index` 向该信号发送信令。<br>门铃信号由 HSA 运行时在队列创建时分配，并且在可用的信号操作上有限制。不支持原子读-改-写操作。 |

- `write_index` - 一个 64 位无符号整数，在队列创建时初始化为 0，指定了应用程序或用户级运行时将要分配的下一个 AQL 数据包槽位的数据包 ID。下一个 AQL 数据包分配在虚拟地址 `base_address + (write_index % size) * AQL数据包大小`。
- `read_index` - 一个 64 位无符号整数，在队列创建时初始化为 0，指定了计算单元硬件将要消费的下一个 AQL 数据包的数据包 ID。尚未被数据包处理器释放的最老 AQL 数据包位于虚拟地址 `base_address + (read_index % size) * AQL数据包大小`。

`write_index` 和 `read_index` 属性不能由 HSA 应用程序或 HSAIL 内核使用内存操作直接访问。相反，它们是通过 HSA 运行时 API（对于 HSA 应用程序）或特定的 HSAIL 指令（如 2.8.5 队列索引访问（第 22 页）所述）来访问的。

### 2.8.3 队列机制

HSA 为如何使用上述定义的信息实现队列规定了以下规则。这些规则定义了提交代理和数据包处理器之间的契约，使提交代理能够以跨 HSA 实现可移植的方式为关联的代理入队工作。

AQL 队列机制的要点如下：

- 代理通过递增 `write_index` 来分配 AQL 数据包槽位。递增操作之前的 `write_index` 值就是所分配的 AQL 数据包的数据包 ID。
- 代理通过将 AQL 数据包的 `format` 字段从 `INVALID` 更改来将数据包指派给数据包处理器。代理**必须**确保在写入 `format` 字段之前或同时，AQL 数据包的其余部分已全局可见。
- 当代理将 AQL 数据包指派给数据包处理器时，AQL 数据包的所有权被转移。数据包处理器可以在数据包提交后的任何时间更新 AQL 数据包，提交代理不应依赖于提交后 AQL 数据包的内容。
- 代理通过使用等于该数据包的数据包 ID 或后续数据包 ID 来向队列门铃发送信令，从而通知数据包处理器。一次队列门铃信号操作可以通知多个 AQL 数据包。应将最后通知的数据包的数据包 ID 用作门铃的值。
- 当队列门铃被发出信令时，要求提交代理确保门铃值所引用的任何 AQL 数据包已经全局可见。不允许使用生产者尚未设置为有效的数据包的数据包 ID 来对门铃发出信令。
- 在小型机器模型平台上，数据包 ID 的最低有效 32 位用于门铃值。
- 对于类型限制为单生产者机制的队列，门铃上发出信令的值**必须**是单调递增的，并且直到通知数据包 ID 的所有数据包**必须**已经被指派。
- 只有数据包处理器或其功能等价物才被允许读取/评估门铃信号。生产 AQL 派发的代理**必须**仅写入门铃信号。除数据包处理器之外的代理从门铃信号读取的值是未定义的。参见第 3 章 HSA 内存一致性模型（第 45 页）。

代理通过执行以下步骤向队列提交任务：

1. 分配一个 AQL 数据包槽位。
2. 用任务细节更新 AQL 数据包（注意，`format` 字段必须保持不变）。
3. 将数据包指派给数据包处理器。
4. 向数据包处理器通知该数据包。

此外，以下详细约束适用：

- HSA 信令机制仅限于其初始化所在的进程地址空间内，参见 2.3 需求：扁平寻址（第 13 页）。
- 在用户模式队列实现中使用此信令机制意味着对用户模式队列有相同的限制；只有队列进程地址空间内的代理才被允许向队列提交 AQL 数据包。
- 队列的大小由 AQL 数据包的数量衡量，且**必须**是 2 的幂。
- `base_address` **必须**始终对齐到 AQL 数据包大小。
- 随着代理向队列添加新的 AQL 数据包，`write_index` 会增加。`write_index` 永远不会回绕，但实际使用的地址偏移量是 `((write_index % size) * AQL数据包大小)`。
- 随着数据包处理器处理并释放 AQL 数据包，`read_index` 会增加。`read_index` 永远不会回绕，但实际使用的地址偏移量是 `((read_index % size) * AQL数据包大小)`。
- AQL 数据包的大小为 64 字节。
- 当 `write_index == read_index` 时，队列为空。
- 当队列创建时，AQL 数据包槽位的 `format` 字段由 HSA 系统初始化为 `INVALID`，并且当数据包已被处理时，也会被设置为 `INVALID`。
- 提交代理**可以**仅在满足 `"packet ID < read_index + size"` 条件且 AQL 数据包的 `format` 字段设置为 `INVALID` 时，修改队列内已分配的 AQL 数据包。
- 数据包处理器**不得**处理 `format` 字段设置为 `INVALID` 的 AQL 数据包。
- AQL 数据包的前 32 位（这包括数据包头，因此也包括 `format` 字段）**必须**仅使用 32 位原子事务来访问。
- 在数据包被通知给数据包处理器之前，数据包处理器不要求处理该 AQL 数据包。
- 数据包处理器可以在数据包 `format` 字段更新之后，但在门铃被发出信令之前处理 AQL 数据包。代理**不得**依赖门铃作为 "门控" 来阻止 AQL 数据包的执行。

队列中的 AQL 数据包按顺序派发，但可以按任意顺序完成。队列中标记为 `INVALID` 的 AQL 数据包将阻塞后续数据包的派发。

数据包处理器**可以**选择在数据包提交后的任何时间释放数据包槽位（通过递增 `read_index` 并将 `format` 字段设置为 `INVALID`），而不管数据包指定的任务状态如何。代理不应依赖 `read_index` 值来推断特定任务的任何状态。

数据包处理器**必须**确保在 `read_index` 移过数据包并使其可用于任务提交之前，数据包槽位的 `format` 字段已被设置为 `INVALID` 并且已全局可见。

每个用户模式队列的用户模式队列基地址和队列大小可能受到系统软件的系统相关限制（例如，队列大小和基地址对齐到系统页大小等等），只要所施加的硬件限制不妨碍 HSA 软件为每个应用程序进程分配多个此类用户模式队列的使用，并且系统软件提供允许在特定系统上施加此类限制的 API。队列由 HSA 应用程序通过 HSA 运行时分配。

数据包处理器**应**对所有并发运行进程的所有用户模式队列具有可见性，并且**应**提供相对于进程的内存隔离，并支持对用户级命令队列中指令的特权限制。与为主机 CPU 指令集定义的应用级特权一致。HSA 系统处理在数据包处理器的可用队列列表之间切换，如 HSA 运行时定义。

HSA 应用程序可以包含代理上多个执行单元的多个集合，这些执行单元可能向同一个 AQL 队列提交作业。在 HSA 运行时的上下文中，HSA 运行时可用于调度多个作业的执行。

为了支持从多个代理提交作业，更新队列的 `write_index` **必须**使用读-改-写原子内存操作。如果仅使用代理上的单个执行单元，则可以使用原子存储而不是读-改-写原子操作来完成更新。

有几种实现多执行单元数据包提交的选项，每种选项在队列已满时避免死锁（即 `write_index >= (read_index + size)`）的特性不同。两个例子是：

- 在递增 `write_index` 之前检查队列是否已满，并对 `write_index` 的递增使用原子 CAS 操作。这允许在队列已满的所有情况下中止数据包槽位分配，但如果没有多个写入者发生冲突，则不需要这样做。
- 首先，原子地将要分配的 AQL 数据包数量加到 `write_index` 上，然后检查队列是否已满。当许多执行单元提交 AQL 数据包时，这可能更有效，但如果队列已满，则可能难以管理队列。

或者，HSA 应用程序可以将作业提交限制为仅一个代理执行单元（"单提交代理"），允许代理和数据包处理器利用此属性，并为此类队列简化作业提交流程。

### 2.8.4 多个 vs. 单个提交代理

对于仅由单个代理提交的队列，可以简化提交流程：

- 提交作业的代理对 `write_index` 值的递增可以使用原子存储完成，因为它是唯一允许更新该值的代理。
- 提交代理可以使用私有变量来保存 `write_index` 值的副本，并且可以假设没有其他人会修改 `write_index`，从而避免读取 `write_index`。

可选地，可以将队列初始化为仅支持单生产者队列机制。某些 HSA 实现可能会为单生产者队列机制实现优化的数据包处理器行为。

某些 HSA 实现可能会对多生产者队列机制和单生产者队列机制的情况实现相同的数据包处理器行为。数据包处理器的行为可以进行优化，只要符合规范即可。例如，在任何一种情况下，都允许数据包处理器：

- 使用数据包的 `format` 字段来确定数据包是否已提交，而不是读取 `write_index` 队列字段。
- 从队列中推测性地读取多个数据包。
- 不为每个处理的数据包更新 `read_index`。

类似地，提交作业的代理不需要为每个提交的数据包读取 `read_index`。

队列机制不要求按顺序提交数据包（除了限制为单生产者机制的队列）。但是，使用门铃，可以在数据包按顺序提交的常见情况下优化数据包处理器的延迟。因此，**建议**从单个执行单元内始终按顺序对门铃发出信令（对于使用单生产者机制的队列，这是严格要求），并避免不同执行单元之间在队列上过度争用。

### 2.8.5 队列索引访问

队列的 `write_index` 和 `read_index` 属性由 HSA 应用程序使用 HSA 运行时 API 调用访问，并由 HSAIL 内核使用特定指令访问。通过 HSA 运行时 API 和特定 HSAIL 指令提供的用于访问队列的 `write_index` 和 `read_index` 属性的操作是等效的。这些操作如下：

- **加载队列的 `write_index` 属性**。返回 64 位的 `write_index` 值，并接受以下参数：
    - 指向队列结构的指针。
- **存储 `write_index` 属性的新值**。无返回值，接受以下参数：
    - 指向队列结构的指针。
    - 要存储在 `write_index` 属性中的新 64 位值。
- **对 `write_index` 属性执行比较并交换操作**。如果原始值与比较参数匹配，则比较并交换操作会更新 `write_index` 属性。返回 `write_index` 属性的原始值，并接受以下参数：
    - 指向队列结构的指针。
    - 64 位比较值。
    - 如果比较值与原始值匹配，要存储在 `write_index` 属性中的新 64 位值。
- **对 `write_index` 属性执行加法操作**。返回 `write_index` 属性的原始值，并接受以下参数：
    - 指向队列结构的指针。
    - 64 位递增值。
- **加载 `read_index` 属性**。返回 64 位的 `read_index` 值，并接受以下参数：
    - 指向队列结构的指针。
- **存储 `read_index` 属性的新值**。无返回值，接受以下参数：
    - 指向队列结构的指针。
    - 要存储在 `read_index` 属性中的新 64 位值。

不同操作需要以下内存排序选项：

- 加载 `read_index` 或 `write_index`：
    - 系统作用域的宽松原子加载
    - 系统作用域的原子加载获取
- 存储 `read_index` 或 `write_index`：
    - 系统作用域的宽松原子存储
    - 系统作用域的原子存储释放
- 对 `write_index` 的加法或比较并交换操作：
    - 系统作用域的宽松原子操作
    - 系统作用域的原子获取操作
    - 系统作用域的原子释放操作
    - 系统作用域的原子获取-释放操作

在内存模型中，所有对队列索引的访问操作的排序方式，就如同队列索引存储在全局段中一样。除主机 CPU 或内核代理之外的其他代理可以通过实现特定的方式访问 `write_index` 和 `read_index` 属性。

### 2.8.6 运行时服务派发队列

HSA 系统使用基于队列的机制，通过向适当的队列发出 `AGENT_DISPATCH` 数据包来调用已定义的系统及 HSA 运行时服务和功能。

此服务队列由应用程序以应用程序特定的方式提供给代理，例如作为（代理）派发数据包的参数，或在已知的内存位置中。

要根据本规范的 2.9.7 代理派发数据包（第 28 页）调用运行时服务或功能，调用者需要在 `AGENT_DISPATCH` 数据包中传递参数和功能代码；调用者可以监视或等待完成信号，以识别 "调用" 何时完成。

在通过 `AGENT_DISPATCH` 的 `completion_signal` 指示完成之前，调用者**不得**假定所服务的功能已启动或完成（参见 2.9.2 数据处理流程（第 26 页））。

当 `AGENT_DISPATCH` 完成时，`AGENT_DISPATCH` 数据包内引用的 `return_address` 将包含该功能的返回值。

`AGENT_DISPATCH` 的派发和数据处理流程**必须**遵循本规范下文 2.9 需求：架构化队列语言 (AQL) 中规定的所有要求。

调用者**必须**遵循所调用功能的所有文档化功能要求，特别是与获取和释放栅栏作用域以及引用的内存相关的要求。

运行时服务队列提供的具体服务由实现运行时服务队列的应用程序定义。

## 2.9 需求：架构化队列语言 (AQL)

兼容 HSA 的系统**应**为代理命令的派发提供一个命令接口。此命令接口由架构化队列语言 (AQL) 提供。

AQL 允许代理构建并使其自己的命令数据包入队，从而实现快速、低功耗的派发。AQL 还为内核代理队列提交提供支持：内核代理内核可以用 AQL 格式编写命令。

AQL 定义了几种不同的数据包类型：

- **供应商特定数据包** - 供应商特定数据包的数据包格式除了 `format` 字段外由供应商定义。对于供应商特定数据包，数据包处理器的行为是实现特定的，但**不得**导致权限提升或脱离进程上下文。
- **无效数据包** - 当队列初始化或 `read_index` 递增使得数据包槽位对代理可用时，数据包格式被设置为无效。
- **内核派发数据包** - 内核派发数据包包含供内核代理执行的作业，并由代理创建。队列的 `features` 字段指示队列是否支持内核派发数据包。
- **代理派发数据包** - 代理派发数据包包含供代理执行的作业，并由代理创建。队列的 `features` 字段指示队列是否支持代理派发数据包。
- **屏障-与 数据包** - 代理可以插入屏障-与数据包以延迟后续数据包的处理。所有队列都支持屏障-与数据包。
- **屏障-或 数据包** - 代理可以插入屏障-或数据包以延迟后续数据包的处理。所有队列都支持屏障-或数据包。

### 2.9.2 数据处理流程

所有数据包类型遵循相同的总体流程。数据处理分为启动、活动和完成阶段：

- **启动阶段**在满足启动条件时开始。启动条件是：
    - 队列中所有前面的数据包必须已完成其启动阶段。
    - 如果数据包头中的 `barrier` 位被设置，则队列中所有前面的数据包必须已完成。
- 在启动阶段，对于内核派发和代理派发数据包，在数据包进入活动阶段之前会应用获取内存栅栏。对于屏障-或 和 屏障-与 数据包，获取内存栅栏稍后在完成阶段应用。
- 在执行获取栅栏（如果适用）之后，启动阶段完成，数据包进入活动阶段。在活动阶段，不同数据包的行为有所不同：
    - 内核派发数据包和代理派发数据包在内核代理/代理上执行，当任务完成时活动阶段结束。
    - 屏障-与 和 屏障-或 数据包保持活动阶段，直到其条件满足。此外，在屏障-与 或 屏障-或 数据包完成其活动和完成两个阶段之前，数据包处理器不会处理任何后续数据包。
- 活动阶段之后是完成阶段。
    - 如果数据包是屏障-与 或 屏障-或 数据包，则作为第一步应用获取内存栅栏。对于内核派发和代理派发数据包，获取内存栅栏已在启动阶段应用。
    - 在执行获取栅栏（如果适用）之后，应用内存释放栅栏。
    - 在内存释放栅栏完成后，AQL 数据包中的 `completion_signal` 字段指定的信号通过递减原子操作被发出信令。`NULL` 是 `completion_signal` 字段的有效值，在这种情况下不执行信令操作。

### 2.9.3 错误处理

如果在数据处理或执行期间检测到错误，则遵循以下顺序：

- 如果错误是在数据包活动执行阶段检测到的，则对失败的数据包应用释放栅栏。
- 该队列将被设置为错误状态，并将通过 HSA 运行时发出回调。可以使用 HSA 运行时 API 查询错误。
- 在遇到错误的队列上，不会启动任何后续数据包，但是任何已经在执行的数据包将继续执行直到完成，或者通过 HSA 运行时中止。
- 同一进程内（相同 PASID）其他队列中的数据包仍将被处理。

### 2.9.4 供应商特定数据包

供应商特定数据包的数据包头中的 `format` 字段设置为 `VENDOR_SPECIFIC`（例如，零）。数据包的所有其他字段是实现特定的。

### 2.9.5 无效数据包

无效数据包的数据包头中的 `format` 字段设置为 `INVALID`（例如，一）。数据包的所有其他字段均为无关项 (don't-care)。

### 2.9.10 小型机器模型

HSA 小型机器模型具有 32 位而非 64 位的原生指针。这会影响任何指定地址的数据包字段：

- 内核派发数据包的 `kernarg_address` 字段
- 代理派发数据包的 `return_address` 字段

对于这些字段，在小型机器模型中仅使用最低有效 32 位。

## 2.10 需求：代理调度

兼容 HSA 的系统要求为每个代理从连接到该代理的 HSA 队列以及任何额外的非 HSA 任务池中调度任务。调度的确切机制是实现定义的，但应该满足本节中概述的限制。

HSA 编程模型鼓励每个应用程序使用多个队列，以便能够调度复杂的任务序列。预期兼容 HSA 的应用程序会同时使用多个队列。为了确保跨 HSA 兼容系统的一致性能，**建议**在使用多个队列时，调度算法能够跨队列高效调度，并最小化管理多个队列的开销。

为确保这一点，代理调度必须具有以下特性：

1.  至少，代理调度**应该**由以下事件触发：
    - 代理任务执行完成，代理可用于执行另一个任务。
    - 如果代理未执行任务，则代理调度**必须**还由以下事件触发：
        - 向连接到代理的任何队列提交数据包。
        - 连接到代理的队列上处于活动状态的 AQL 屏障-与 或 屏障-或 数据包所使用的信号被发出信令，使得该 AQL 屏障-与 或 屏障-或 数据包可以完成。
    - 代理调度**必须**至少能够监视连接到它的每个队列上处于最前端的活动 AQL 数据包。监视包括任何受支持的 AQL 数据包及其依赖项的组合。为了确保避免死锁，对依赖项的监视**不得**占用任何执行任务前进保证所需的执行资源（例如，一个队列上活动的屏障-与 或 屏障-或 数据包**不得**阻塞其他队列上的任务执行）。

2.  代理调度**应该**在有限时间内响应第 1 点中列出的事件。

3.  当代理调度机制被触发并且代理有可用的执行资源时，如果连接到代理的任何队列上最老的数据包是一个可运行的内核派发或代理派发数据包，则调度机制**必须**派发一个任务。注意，屏障-与 和 屏障-或 数据包不允许占用代理/内核代理的执行资源。

4.  支持完整配置文件的代理**必须**表现出至少支持每个已连接队列的最老数据包同时处于活动阶段（参见 2.9.2 数据处理流程（第 26 页））。这可以通过使用代理的上下文切换来为跨多个队列的活动内核派发或代理派发数据包分配时间片来实现。

5.  支持基础配置文件的代理要求至少支持系统中一个已连接队列的最老数据包处于活动阶段（参见 2.9.2 数据处理流程（第 26 页）），但不要求跨多个队列提供独立的前进保证。

## 2.11 需求：内核派发前进保证

以下规则适用于内核派发所需的前进保证：

- 对于完整配置文件代理，对于每个已连接队列的最老活动派发，具有最低扁平化工作组 ID 的不完整工作组**必须**是活动的。
- 对于基础配置文件代理，对于一个已连接队列的最老活动派发，具有最低扁平化工作组 ID 的不完整工作组**必须**是活动的。
- 工作组在启动时或挂起的工作组恢复时变为活动状态。
- 工作组在完成或挂起时停止活动。
- 当波前所属的工作组是活动的时候，该波前是活动的。
- **完成**
    - 波前在其所有工作项都执行完最后一条指令后完成。
    - 工作组在工作组的所有波前都完成后完成。
    - 内核派发的活动阶段（参见 2.9.2 数据处理流程（第 26 页））在内核派发的所有工作组都完成后完成。
- 在活动的工作组的波前中，至少有一个非等待波前**必须**取得前进保证。
- 波前处于等待状态，如果：
    - 正在执行未满足的 `barrier`（屏障）指令
    - 正在执行未满足的 `fbarrier`（栅栏屏障）等待
    - 任何通道正在执行未满足的信号等待
- 波前取得前进保证，如果：
    - 至少有一个工作项正在执行指令

处于活动阶段的内核派发的活动工作组在其执行期间可能会因代理调度和负载均衡而发生变化。工作组启动、完成、挂起或恢复的确切顺序，以及随之产生的活动工作组的数量，被认为是平台特定的，不由 HSA 规范强制规定；但是，对于处于活动阶段的派发，上文定义了活动工作组的最小数量。

对于完整和基础配置文件，在 2.10 需求：代理调度（第 30 页）中定义了哪些派发必须处于活动状态的要求。

扁平化工作组 ID 和波前发散/收敛在 HSA 程序员参考手册 版本 1.1 中定义。

## 2.12 需求：IEEE754-2008 浮点异常

内核代理**应**向 HSA 运行时报告与执行 HSAIL 代码相关的某些已定义异常。至少**应**报告以下 IEEE754-2008 浮点错误异常：

**表 2-11 IEEE754-2008 异常**

| 操作 | 异常代码 | 描述 |
| :--- | :--- | :--- |
| INVALID_OPERATION | 0 | 无效操作，在默认异常处理下会引发标志，如 IEEE754-2008 中所规定。 |
| DIVIDE_BY_ZERO | 1 | 除零，在默认异常处理下会引发标志，如 IEEE754-2008 中所规定。 |
| OVERFLOW | 2 | 溢出，在默认异常处理下会引发标志，如 IEEE754-2008 中所规定。 |

**DETECT (检测):**

基础 HSA 配置文件中不要求支持 DETECT。完整 HSA 配置文件中要求支持 DETECT。

内核代理**必须**为每个工作组维护每个启用 DETECT 的算术异常的状态位。状态位在派发开始时重置为零。如果发生异常并且为该异常启用了 DETECT 策略，则内核代理将设置相应的状态位。内核代理**必须**：

- 支持能在内核运行时读取或写入异常状态字段的 HSAIL 指令。

**BREAK (中断):**

基础 HSA 配置文件中不要求支持 BREAK，完整 HSA 配置文件中可选支持。

当发生异常并为该异常启用了 BREAK 策略时，内核代理**必须**在产生异常的指令处精确停止异常波前的 HSAIL 指令流执行。内存和寄存器状态刚好在产生异常的指令执行之前，除了产生异常的波前中未发生异常的通道可能已经执行并更新了寄存器或内存状态。在合理的时间内，执行该波前的内核代理**必须**停止为在同一用户模式队列上执行的所有派发启动新的波前，并且**必须**确保当前正在为这些派发执行的所有波前要么完成，要么被停止。任何完成的派发将正常更新其完成信号，但是，任何未完成执行其所有关联波前的派发将不会更新其完成信号。

然后，内核代理会向运行时发出信令，运行时将调用 HSA 运行时中定义的异常处理程序。报告的信息**必须**包括：

- 异常代码
- PASID
- 队列 ID
- 数据包 ID
- 导致异常的精确指令计数器。

运行时异常处理程序的一些可能用途是：

- 调用调试器，允许开发者确定导致异常的函数并检查周围的内存和寄存器状态。
- 将异常记录到文件。
- 终止进程。

在发出 BREAK 信令后，不要求内核代理能够重新启动产生异常的指令。

DETECT 和 BREAK 异常策略的启用通过 HSAIL 最终化进行控制。可以为同一个算术异常同时启用 DETECT 和 BREAK。

异常处理策略必须在 HSAIL 代码最终化时指定，因为当请求异常处理时，实现可能会生成不同的代码。启用 DETECT 或 BREAK 功能可能会降低在内核代理上运行的内核的性能。但是，**建议**启用 DETECT 策略的性能接近原生代码性能，因为这是一个常用的异常策略。

## 2.13 需求：内核代理硬件调试基础设施

内核代理**应**提供机制，以允许系统软件和某些选定的应用软件（例如，调试器和性能分析器）设置断点并收集吞吐量信息以进行性能分析。至少**应**支持指令断点。内存断点、内存故障和条件断点可以作为实现选项来支持。

当命中断点条件时，硬件**应**将所有挂起的写入提交到内存，停止波前，并提供状态信息，指示断点条件、受影响的 PASID、用户级队列、指令以及波前/组 ID。

系统软件**应**提供适当的常驻内存，以允许内核代理硬件在达到断点条件时写出状态信息，该信息可由相应的选定应用软件（例如，调试器和性能分析器）解析。写入数据的格式**应**是系统实现特定的。

## 2.14 需求：HSA 平台拓扑发现

### 2.14.1 引言

HSA 系统架构需求涵盖了具有各种元素的各种系统。为了充分利用可用元素，应用和系统软件需要一种方法来发现和描述可用的代理、它们彼此之间以及与系统中其他相关资源（如内存、缓存、I/O 和主机代理（例如，CPU））的拓扑关系。

兼容 HSA 的系统包含各种不同的代理、接口和其他资源，例如内存和缓存。应用、运行时和系统软件需要一种方法来描述所有可用资源以便使用它们。

至少，兼容 HSA 的系统必须提供一组可供 HSA 软件使用的参数特征描述。应用程序可以通过各种运行时 API（通过 HSA 运行时，预期其他面向平台的 API，例如 OpenCL™ 和 DirectCompute，会报告等效信息）来发现这些参数，并且软件可以利用这些参数来优化在特定平台上的性能。

除非系统软件以其他方式提供给 HSA 运行时级别，否则这些属性中的每一个**应**通过适当的参数存储检索来提供；实现可以使用固件表项、通过固件方法检索的数据（例如，通过 ACPI）、内核镜像或通过等效机制。众多内核代理属性可以被枚举，它们彼此之间的关系可以通过解析数据的系统软件建立，并传递给 HSA 运行时和 HSA 应用程序。

根据平台的不同，数据**应该**按层次结构分组，以对应于诸如代理、内存、计算属性、缓存和 I/O 等组件。

数据与节点 ID 相关联，形成一个按节点的元素列表，该列表引用了该列表内相对偏移处包含的元素。一个节点与一个内核代理或代理相关联。节点 ID **应该**是从 0 开始的，"0" ID 代表系统的主元素（例如，"引导核心"、内存）（如果适用）。枚举顺序以及（如果适用）ID 值**应该**与通过 HSA 需求范围外的其他机制报告的信息相匹配；例如，某些系统上 ACPI SRAT 表中包含的数据和枚举顺序**应该**与通过 HSA 报告的内存顺序和属性相匹配。更多细节超出了系统架构的范围，并在运行时 API 规范中概述。

这些节点中的每一个在更高级的系统中都与其他节点互连，其互连程度足以充分描述 HSA 系统拓扑。

在适用的情况下，物理内存的节点分组遵循 NUMA 原则，以便在系统中存在多个物理内存块且代理对该内存具有不同的 "访问成本"（例如，带宽/延迟）时，软件可以利用内存局部性。

### 2.14.2 拓扑需求

平台拓扑是 HSA 系统的固有属性。从应用程序的角度来看，拓扑可以被完全枚举，并且在应用程序枚举过程本身期间不应更改，但应用程序之后可能会收到更改通知。

对于支持对接或动态连接/移除组件（"对接"）的高级系统，系统必须提供属性来检测短暂组件与持久组件，并使用通知机制告知 HSA 运行时和应用程序，允许它们重新枚举更新后的拓扑。在枚举期间检测到的拓扑更改应推迟到先前拓扑的枚举完成后，以避免应用程序数据结构中出现未定义或瞬态的拓扑状态。

通常，但并非在所有情况下，系统软件可以在系统启动时或接近启动时枚举所有 HSA 系统拓扑信息，并在应用程序启动时使其可用。系统软件和运行时可以为应用程序提供机制，以通知其可能由平台和系统事件（例如，对接）引起的拓扑更改，这些更改会改变先前报告的拓扑。

详细的通知机制可能是系统和供应商特定的，超出了本规范的范围。

每个数据参数描述了系统中特定内核代理的属性。此外，代理条目可以引用描述内核代理所关联资源的元素。每个拓扑元素由唯一的元素 ID 和特定于类型的属性数据来表征。

目标是允许兼容 HSA 的系统软件和 HSA 应用程序识别层次结构中的执行块，在代理/内核代理内部，以便单独调度和寻址；例如，在 PC 平台中，代理（如 CPU）可以以诸如 "插槽" 和核心的单元来表达，每个插槽的内存访问路径和缓存层次结构代表一个 "节点"，而核心代表可单独调度的执行元素，表示为代理。

至少，以下代理/内核代理参数**应该**对系统软件和通过应用程序编程接口 (API)（例如，HSA 运行时，OpenCL™，或其他）对应用程序可用：

- "内核代理" 或 "代理" 指示符（见上文）
- 类型（吞吐量/向量，延迟/标量，DSP，...）
- 元素所属的节点 ID。
- 元素 ID。用于在平台上唯一标识该代理。平台定义中固有的现有值可用于此目的。不要求顺序性。
- 最大计算单元数量
- 最大工作项维度
- 最大工作项大小
- 机器模型
- 端序
- 最大和最小队列大小，始终是 2 的幂，且是 AQL 数据包大小的倍数

代理的可选属性也将通过此机制公开，例如：

- 最大时钟频率（MHz）（如果未在平台上公开，则为 0）
- 友好名称（等效于 OpenCL™ 中的 "设备名称"）
- 浮点能力

其他可选参数可能取决于平台。平台未定义的任何可选参数应返回为 "0"。

### 2.14.4 内存条目

HSA 内存条目描述了系统中物理内存块的物理属性，这些内存块具有可识别的特性，并且可以被代理通过 HSA 内存模型语义访问。

至少，以下每个物理上可区分的内存块的 HSA 内存参数将对系统软件可用，并可能通过 API（例如，HSA 运行时，OpenCL™，或其他）对应用程序可用：

- 元素所属的节点 ID。
- 元素 ID。用于在平台上唯一标识该内存块。平台定义中固有的现有值可用于此目的。不要求顺序性。
- 虚拟基地址（32 位或 64 位值）
- 物理内存大小（以字节为单位）

可选的 HSA 内存参数也将通过此机制公开，例如：

- 内存交错（1, 2, 4）
- 内存通道数量
- 每个内存通道的内存宽度
- 最大内存吞吐量（MByte/s）
- 最小内存访问延迟（皮秒）

其他可选参数可能取决于平台。平台未定义的任何可选参数应返回为 "0"。

### 2.14.5 缓存条目

HSA 缓存条目描述了系统中代理可用的缓存。

至少，以下 HSA 内存参数将对系统软件可用，并可能通过 API（例如，HSA 运行时，OpenCL™，或其他）对应用程序可用：

- 元素所属的节点 ID。
- 元素 ID。用于在平台上唯一标识该内存块。平台定义中固有的现有值可用于此目的。不要求顺序性。
- 缓存类型，例如，只读或读/写缓存
- 缓存总大小（千字节）
- 缓存行大小（字节）
- 缓存级别（1, 2, 3, ...）
- 关联性（0 = 保留，1 = 直接映射，2 - 0xFE = 相应路数的组相联，0xFF = 全相联）
- 相对于较低级别缓存的独占/包含性
- 如果缓存在代理之间共享，则关联的代理（通过 `nodemask` 标识）

可选的 HSA 内存参数也将通过此机制公开，例如：

- 缓存命中的最大延迟（皮秒）

其他可选参数可能取决于平台。平台未定义的任何可选参数应返回为 "0"。

## 2.15 需求：图像

兼容 HSA 的平台**应**可选地提供能力，使 HSA 软件能够定义和使用图像对象，这些对象用于存储一维、二维或三维图像。

图像对象的元素是根据预定义的架构化或供应商特定的图像几何、格式和数据布局列表来定义的。图像原语可以从内核代理访问，并操作图像值，该值使用不透明的 64 位图像句柄引用。

图像句柄由 HSA 运行时创建。图像对象可以是只读的（即内核代理只能从图像读取值），只写的（即内核代理只能向图像写入值），或者可读写的（即内核代理可以写入图像也可以从图像读取）。

内核代理**可以**选择不支持图像对象。

图像几何指的是图像中元素的逻辑组织，例如，图像的维数以及它是否为图像数组。

图像大小指的是每个图像维度的大小以及图像数组的层数。

图像格式指的是图像中每个元素的逻辑组织，例如，图像中每个元素的通道分量的数量、顺序和类型。

图像数据布局指的是图像在内存中的存储方式。

如果支持图像，则对图像几何、格式和数据布局的支持有强制性要求：

- **必须**支持一组强制性的图像几何和格式，但图像数据布局可以是实现特定的（不透明）。强制性的图像几何和格式在 HSA 程序员参考手册中进一步详细说明。此外，"1DB" 图像格式**必须**支持 "线性" 图像数据布局。"1DB" 图像格式和 "线性" 数据布局在 HSA 程序员参考手册中进一步描述。

图像具有以下附加属性：

- 图像对象由 HSA 平台通过运行时调用创建，并带有一个指定的全局段内存缓冲区用于图像数据存储。
- 图像对象**可以**仅由其创建时所针对的代理使用。
- 同一个全局段内存缓冲区**可以**用于为多个代理创建图像对象。
- 针对同一个代理的多个图像对象**可以**别名到同一个图像存储区，如果所选图像数据布局支持不同的访问模式，则这可用于将只读、只写和读写图像彼此别名。
- 允许对图像数据存储进行所有权指派操作。请注意，图像数据存储所有权的更改被视为图像数据的修改，并且必须根据下文定义的图像数据可见性规则执行。
- 图像对象的图像数据存储的数据布局**必须**是创建该图像对象时代理所支持的架构化或供应商特定图像数据布局之一。
- 支持一个可选的架构化不透明图像数据布局。不透明图像数据布局具有以下属性：
    - 不保证不透明图像数据布局在不同的代理之间是一致的。当使用不透明图像数据布局时，图像数据存储不能在针对不同代理的图像对象之间共享。
    - 保证不透明图像数据布局在单个代理上正确别名到同一图像存储区的所有图像对象之间是一致的。多个图像对象必须满足才能正确别名到同一图像存储区的图像对象属性约束在 HSA 程序员参考手册中描述。

图像操作不影响 HSA 内存模型定义的内存排序。图像操作可见性适用以下规则：

- 通过 HSA 运行时 API 对图像数据的修改要求发生以下顺序，才能对后续的 AQL 数据包派发可见：
    1.  修改图像数据的 HSA 运行时操作完成。HSA 运行时操作意味着对图像数据的释放操作。
    2.  在读取图像的代理上执行适用于图像数据的数据包处理器获取栅栏（这可以是读取修改后图像数据的 AQL 数据包派发的一部分），并与先前的隐式释放操作同步。
- 代理通过内存操作对图像数据存储的修改要求发生以下顺序，才能对后续的 AQL 数据包派发可见：
    1.  在图像数据存储修改完成后，执行修改的代理使用适当的作用域执行释放操作。
    2.  在读取图像的代理上执行适用于图像数据的数据包处理器获取栅栏（这可以是读取修改后图像数据的 AQL 数据包派发的一部分），并与先前的释放操作同步。
- 内核代理数据包派发对图像数据的修改可见性如下：
    - 通过图像操作进行的 AQL 数据包派发对图像数据的修改要求发生以下顺序，才能对后续的 AQL 数据包派发可见：
        1.  修改图像的 AQL 数据包派发的活动阶段完成。
        2.  在修改图像的代理上执行适用于图像数据的数据包处理器释放栅栏（这可以是修改图像的 AQL 数据包派发的一部分）。
        3.  在读取图像的代理上执行适用于图像数据的数据包处理器获取栅栏（这可以是读取修改后图像数据的 AQL 数据包派发的一部分），并与先前的 AQL 数据包释放栅栏同步。
    - 通过图像操作进行的 AQL 数据包派发对图像数据的修改要求发生以下顺序，才能对 HSA 运行时操作可见：
        1.  修改图像的 AQL 数据包派发的活动阶段完成。
        2.  在图像的代理上执行适用于图像数据的数据包处理器释放栅栏（这可以是修改图像的 AQL 数据包派发的一部分）。
        3.  执行 HSA 运行时操作。HSA 运行时操作意味着对图像数据的获取操作，并且必须与先前的 AQL 数据包释放栅栏同步。
    - 通过图像操作进行的 AQL 数据包派发对图像数据的修改要求发生以下顺序，才能对代理的内存操作可见：
        1.  修改图像的 AQL 数据包派发的活动阶段完成。
        2.  在修改图像的代理上执行适用于图像数据的数据包处理器释放栅栏（这可以是修改图像的 AQL 派发的一部分）。
        3.  在读取图像的代理上执行获取操作，并与先前的 AQL 数据包释放栅栏同步。
- 单个工作项的图像存储操作在执行工作项作用域图像栅栏操作后，对同一工作项的图像加载操作可见。
- 在写入和读取工作项都执行工作组执行统一图像获取/释放栅栏（工作组作用域）之后，工作项的图像存储操作对同一工作项和同一工作组中的其他工作项的图像加载操作可见。
- 图像栅栏操作不影响只读图像，可以在只读图像的图像读取和只读图像的任何修改之间指定排序的唯一栅栏是使用只读图像的代理上的数据包处理器获取栅栏以及一个适当的同步释放栅栏。
- 请注意，在代理作用域内没有图像获取/释放栅栏。因此，无法使一个工作项执行的图像存储对同一派发的不同工作组中的另一个工作项执行的图像加载可见。
- 内核代理在图像操作和内存操作之间指定排序的唯一机制是数据包处理器栅栏。
- 图像栅栏和 barrier/wavebarrier 操作不能被重新排序。barrier/wavebarrier 操作和图像栅栏之间的排序对参与 barrier/wavebarrier 操作的所有工作项可见。有关 barrier/wavebarrier 操作的更多信息，请参见 HSA 程序员参考手册 版本 1.1。
- 单个工作项的图像访问和图像栅栏不能被重新排序。
- 访问尚未可见的修改后图像数据会产生未定义的值。
- 使用至少一个未定义坐标的图像加载和存储会导致程序未定义。

如上所述，图像操作不包含在第 3 章 HSA 内存一致性模型（第 45 页）中定义的 HSA 内存模型中。只有遵循上述可见性规则时，图像操作和内存模型之间才能进行互操作。

## 2.16 需求：性能分析

兼容 HSA 的平台**应**可选地提供对 HSA 软件性能进行分析的能力。

### 2.16.1 性能分析事件扩展

兼容 HSA 的平台**应**可选地提供能力，使 HSA 软件能够定义、产生和读取时间线事件。事件可以由 HSA 运行时 API、HSA 拓扑节点、调用 HSA 运行时的应用程序以及 HSAIL 内核产生。事件类型可以通过运行时 API 注册，并从调用应用程序或 HSAIL 内核触发，并带有可选的元数据。事件具有以下附加要求：

- HSA 平台必须能够向请求事件的应用程序提供单一事件流。为了实现这一点，它可能需要将来自多个源的事件合并到单个缓冲区中，或者平衡来自不同源的事件的提供。不要求事件必须按照它们生成的顺序提供给应用程序。必须有一种方法，使得由 HSAIL 产生的事件及其关联的任何元数据能够被提供给 HSA 运行时 API。

- HSA 平台必须能够告知在任何给定时间可以计数哪些性能计数器集。没有一组必须可用的必需性能计数器；实现可以公开它想要的任何计数器，并具有实现定义的语义。

# 第 3 章. HSA 内存一致性模型

## 3.1 什么是内存一致性模型？

内存一致性模型定义了一个执行单元的写入如何对另一个执行单元变得可见，无论它们是在同一个代理上还是在不同的代理上。本章描述的内存一致性模型设定了一个最低标准。一些 HSAIL 实现可能提供比要求更强的保证。

对于许多实现来说，如果允许硬件或最终化器放宽内存操作执行的感知顺序，将会获得更好的性能。例如，最终化器可能会发现将写入移动到程序后面更高效；只要程序语义不变，最终化器就可以自由地这样做。一旦存储被延迟，其他工作项和代理在存储实际发生之前将看不到它。硬件可能提供一个也会延迟写入的缓存。

同样，硬件和最终化器有时都被允许将写入内存的操作移动到程序更早的位置。在这种情况下，其他工作项和代理可能会在写入发生时（早于它在源代码中似乎被执行的时间）看到它。

内存一致性模型为两类问题提供了答案。对于开发者和工具来说，问题是："我如何让它总能得到正确的结果？" 对于实现来说，问题是："如果我这样做，是允许的吗？"

程序无法直接看到任何全局顺序。实现无法知道未来的加载和存储。每个实现都创建一个顺序，使得未来加载和存储所创建的任何可能的可见性顺序要求都不会被违反。

实现可以比模型允许的更严格地约束事件。该模型是事件的最低要求排序。该模型从操作的描述，到单个计算单元引起的操作的最低要求排序，再到来自多个计算单元的操作在时间上的交织以及允许的重新排序。

合法的结果由交织、允许的重新排序和可见性规则确定。

## 3.2 什么是 HSA 内存一致性模型？

HSA 内存一致性模型描述了多个工作项如何与自身以及与代理进行交互。

HSA 内存一致性模型是一个基于无数据竞争框架的宽松模型。

该内存模型定义了明确定义和未定义执行的概念，并且如果一个程序有任何未定义的执行，则该程序具有未定义的行为。

任何不包含宽松原子操作的 HSA 无竞争程序的执行看起来是顺序一致的。

任何包含一个或多个宽松原子操作的 HSA 无竞争程序的执行可能看起来不是顺序一致的。

HSA 包含内存段和作用域的概念。

每个加载或存储操作都发生在单个内存段上，而栅栏则跨多个段应用。对于特殊的内存操作，我们定义了它适用的一组作用域（即工作项、波前、工作组、代理和系统）。

内存一致性模型的确切规则和定义在 HSA 内存一致性模型部分的其余部分中描述。

系统中可能存在不遵循 HSA 内存一致性模型但可以将数据移入和移出 HSA 内存区域的设备。

软件只能看到加载和读-改-写操作的结果。每个加载结果都创建了一个更受约束的部分可见性顺序。当某个加载获得的结果未能满足系统中所有必需的部分可见性顺序时，契约被破坏。

## 3.3 HSA 内存一致性模型定义

HSA 内存一致性模型是根据执行单元和操作的可见性排序来定义的。

**执行单元** 是通过一个处理元素的程序排序操作序列。执行单元可以是代理上的任何执行线程、一个工作项，或者是通过 HSA 兼容设备中的处理元素发送操作的任何方法。

- 对于类型为 "存储" 的原语，存储操作 X 对执行单元 A 的**可见性**是指存储 X 的数据对于来自执行单元 A 的加载可用。
- 对于类型为 "加载" 的原语，**可见性**是指加载获得了该执行单元将放入寄存器的数据。
- 类型为 "栅栏" 的原语可以对加载和存储操作的可见性施加额外的约束。栅栏在栅栏隐含的约束得到满足的那一点上是**可见的**。例如，一个释放栅栏在受栅栏影响的执行单元的所有先前事务在所需的作用域实例中可见，并且相对于其他栅栏和原子操作的约束也得到满足时，即是可见的。
- **单副本原子属性定义**。如果两个单副本原子存储重叠，则其中一个存储的所有写入将在另一个存储的任何写入被插入之前，被插入到每个重叠字节的一致性顺序中（参见 3.7 一致性顺序（第 52 页））。如果单副本原子加载 A 与单副本原子存储 B 重叠，并且对于任何重叠的字节，加载返回了由单副本原子存储 B 插入到该字节一致性顺序中的写入数据，则加载 A 返回的数据必须来自每个重叠字节的一致性顺序中不早于单副本原子存储 B 插入到该字节的写入的那个点。
- **"未定义的程序或执行"**。某些不正确的程序可能表现出实现特定的行为。这些程序或这些程序的执行通常被标记为 "未定义"。然而，"未定义程序" 的可能行为不包括脱离进程隔离或提升进程的特权级别。未定义程序可能表现出的行为必须与其特权级别一致。未定义程序的一些允许行为的例子是：以不正确的结果完成、发生段违规故障或其他故障等等。

### 3.3.1 操作

"内存操作" 是指任何隐含执行一个或多个加载、存储或栅栏原语的操作。定义了以下基本操作：

- `ld`，加载一个值，也称为 "普通加载"。
- `st`，存储一个值，也称为 "普通存储"。
- `atomic_op`，执行一个返回值的原子操作 `op`。这可以是一个加载 (`ld`) 或一个读-改-写操作 (`rmw`)。
- `atomicNoRet_op`，执行一个不返回值的原子操作。这可以是一个存储 (`st`) 或一个读-改-写操作 (`rmw`)。
- `fence`，执行一个内存栅栏操作。

内存操作顺序约束附加到内存操作上。定义了以下顺序约束：

- `rlx`，无附加顺序约束（即宽松）。
- `scacq`，附加获取约束 \(^1\)。
- `screl`，附加释放约束 \(^2\)。
- `scar`，附加获取和释放约束 \(^3\)。

普通加载或存储允许的唯一顺序约束是 `rlx`。

允许以下原子操作和顺序约束的组合：

- `atomic_ld_rlx`, `atomic_ld_scacq`,
- `atomic_rmw_rlx`, `atomic_rmw_scacq`, `atomic_rmw_screl`, `atomic_rmw_scar`

允许以下原子概念操作和顺序约束的组合：

- `atomicnoret_st_rlx`, `atomicnoret_st_screl`,
- `atomicnoret_rmw_rlx`, `atomicnoret_rmw_scacq`, `atomicnoret_rmw_screl`, `atomicnoret_rmw_scar`

允许以下栅栏操作和顺序约束的组合：`fence_scacq`, `fence_screl`, `fence_scar`。

- **原子存储** 是任何隐含修改内存位置的原子操作。
- **原子加载** 是任何隐含读取内存位置的原子操作。
- **原子读-改-写** 是任何既隐含读取又隐含修改内存位置的操作。
- **普通内存操作** 是普通加载或普通存储 (`ld` 或 `st`)。
- **释放操作** 是所有具有 `screl` 或 `scar` 顺序的内存操作（原子或栅栏）。
- **获取操作** 是所有具有 `scacq` 或 `scar` 顺序的内存操作（原子或栅栏）。
- **同步操作** 是所有释放操作和所有获取操作的并集，即所有具有 `scacq`、`screl` 或 `scar` 顺序的操作（原子或栅栏）。
- **特殊操作** 是所有原子操作和所有栅栏操作。

HSAIL 中可用的操作具有不同的名称，并且可能对操作、顺序、段和作用域的允许组合有限制。有关 HSAIL 内存操作的完整描述，请参见 HSA 程序员参考手册 版本 1.1。

### 3.3.2 原子操作

以下属性适用于原子操作：

- 原子操作**必须**使用自然对齐的地址。如果用于原子操作的地址不是自然对齐的，则该操作是未定义的。
- 原子加载和存储是单副本原子的。
- 原子读-改-写操作在所访问字节的一致性顺序中是不可分割的（参见 3.7 一致性顺序（第 52 页））。写操作在读操作之后立即在字节的一致性顺序中进行。读-改-写操作的读取和写入组件是单副本原子的。
- 同步原子操作（即，具有释放和获取语义的配对原子操作）**必须**针对相同的位置并且具有相同的大小。

### 3.3.3 段

HSA 内存一致性模型定义了以下不同的段：

- **全局段**，在所有代理之间共享。
- **组段**，在 HSAIL 内核派发中同一工作组的工作项之间共享。
- **私有段**，对于 HSAIL 内核派发中的单个工作项是私有的。
- **Kernarg 段**，对于 HSAIL 内核派发中的所有工作项可见的只读内存。
- **Readonly 段**，对所有代理可见的只读内存。
- **Flat 段** 被视为一个虚拟段，所有对 flat 段的操作都被认为是在特定 flat 地址所映射到的实际段上执行的。

只读段在 HSAIL 内核派发之后不会被更新，因此只有普通加载操作与这些段相关。程序可以假设加载将读取内核派发之前某个位置的初始值。只读段在 HSA 内存一致性模型部分不再进一步讨论。

不同的段在 HSA 程序员参考手册 版本 1.1 中有更详细的描述。一个特定的内存位置始终使用相同的段来访问。

### 3.3.4 所有权

**注意：当前的所有权定义已弃用，并可能在下一版本中以不兼容的方式更改。**

可以赋予代理对全局段中某个地址范围的所有权。以下规则适用于所有权：

- 一次只能有一个代理拥有一个地址范围的所有权。
- 当一个代理拥有一个地址范围的所有权时，只允许该代理访问该地址范围。
- 如果代理访问了另一个代理拥有的地址，则程序是未定义的。
- 有两种类型的所有权：
    - **只读所有权**，代理只能从拥有的地址范围读取。
    - **读写所有权**，代理可以写入以及读取拥有的地址范围。
- 地址范围所有权通过 HSA 运行时调用来控制。

### 3.3.5 作用域

HSA 内存一致性模型定义了以下不同的作用域：

- 工作项 (wi)
- 波前 (wave)
- 工作组 (wg)
- 代理 (agent)
- 系统 (system)

内存操作作用域用于将内存操作所需的可见性限制为系统中观察者（执行单元）的子集。

特殊操作指定一个作用域，但是通过包含性可能会隐含多个作用域（参见 3.3.6 作用域实例（下方））。

普通内存操作具有隐式的**工作项**作用域，并且仅在涉及同步操作时才需要对其他执行单元可见，此时它们需要在同步操作的作用域实例内可见。

### 3.3.6 作用域实例

静态程序中命名的作用域对应于执行期间特定的作用域实例。例如，一个指定工作组作用域的特殊操作将对应于一个作用域实例，该实例包含与执行该特殊操作的执行单元在同一工作组中的所有执行单元。

特殊操作的实际作用域受到段的最大可见性的限制。

- 使用 flat 地址执行的特殊操作受限于该地址的底层段映射的最大作用域可见性。
- 在由代理拥有的地址范围内，对全局段中的字节位置执行的、具有系统作用域的特殊操作，其行为如同指定了代理作用域。
- 对组段中的字节位置执行的、具有系统或代理作用域的特殊操作，其行为如同指定了工作组作用域。
- 对组段执行的具有系统或代理作用域的同步操作，将以工作组作用域执行。
- 对私有段执行的具有系统、代理、工作组或波前作用域的同步操作，将以工作项作用域执行。因此，对私有段位置的原子操作没有排序副作用，并且与普通操作类似处理。
- 除了全局段和组段之外的其他段，要么不可写，要么只对单个执行单元可见，因此作用域不适用。

当使用系统以外的其他作用域时，可能存在多个作用域实例。例如，当使用工作组作用域时，不同工作组中的执行单元将指定工作组作用域的不同实例。

作用域实例是包含性的。当指定一个作用域实例时，它隐含了所有更小的作用域实例。例如，如果为某个操作指定了代理作用域，那么也隐含了工作组、波前和工作项作用域实例。

我们定义一个操作符 `SI`，它返回一个内存操作指定的作用域实例集合。

两个不同执行单元执行的两个操作所指定的作用域实例集合 `S1` 和 `S2`，如果它们包含任何公共的作用域实例，则**匹配**。这将在以下情况下发生：

- `S1` 和 `S2` 的最大作用域都是系统，或
- `S1` 或 `S2` 的最大作用域是系统，另一个是代理，并且这些执行单元属于同一个代理，或
- `S1` 或 `S2` 的最大作用域是系统，另一个是工作组，并且这些执行单元属于同一个工作组，或
- `S1` 或 `S2` 的最大作用域是系统，另一个是波前，并且这些执行单元属于同一个波前，或
- `S1` 和 `S2` 的最大作用域都是代理，并且这些执行单元属于同一个代理，或
- `S1` 或 `S2` 的最大作用域是代理，另一个是工作组，并且这些执行单元属于同一个工作组，或
- `S1` 或 `S2` 的最大作用域是代理，另一个是波前，并且这些执行单元属于同一个波前，或
- `S1` 和 `S2` 的最大作用域都是工作组，并且这些执行单元属于同一个工作组，或
- `S1` 或 `S2` 的最大作用域是工作组，另一个是波前，并且这些执行单元属于同一个波前，或
- `S1` 和 `S2` 的最大作用域都是波前，并且这些执行单元属于同一个波前，或
- 这些操作由同一个执行单元执行。

我们定义一个操作符 `Match`，当作用域实例匹配时为真：
`Match(S1, S2) = (∅≠(S1 ∩S2))`

请注意，对于由同一个执行单元（即工作项）执行的操作，始终存在一个公共的作用域实例，因此 `Match` 始终为真。

例如，如果来自不同工作组的工作项使用工作组作用域访问全局段，则它们没有匹配的作用域实例，并且 `Match` 为假。

### 3.3.7 数据包处理器栅栏

此外，内存模型定义了仅由数据包处理器使用的数据包内存栅栏（参见 2.9.1 数据包头（第 25 页））：

- **数据包内存释放栅栏**使得由任何已完成活动阶段的派发（该派发属于同一代理的任何队列）的任何执行单元存储的任何全局段或图像数据，在数据包释放栅栏指定的所有作用域中可见。
- **数据包内存获取栅栏**确保由任何尚未进入活动阶段的派发（该派发属于同一代理的任何队列）的任何执行单元进行的任何后续全局段或图像加载，能够看到先前在数据包获取栅栏指定的作用域中释放的任何数据。

### 3.3.8 特殊操作的前进保证

特殊操作必须取得前进保证，并且必须在有限时间内对其指定的作用域实例可见。不要求普通存储必须在有限时间内可见，但如果它们被排序在某个必须在有限时间内可见的东西之前，那么根据排序行为，这些普通存储将在有限时间内可见，这是一个结果。

## 3.4 可能执行

我们将**可能执行**定义为 HSA 程序的结果，其中每个加载操作观察到在同一执行中发生的任何存储操作的值。可能执行是当 HSA 无竞争程序在 HSA 兼容系统中运行时可能发生的合法执行的超集。

## 3.5 候选执行

候选执行是可能执行的一个子集，其中存在一组必需的可观察操作（普通和特殊）的明显顺序（定义如下），并且这些顺序是一致的。

如果 HSA 程序是明确定义的（参见 3.12 无竞争程序的语义（第 55 页）），那么该程序的实际执行将是候选执行之一。

### 3.5.1 顺序

顺序定义了对一组操作的关系。

\(X >_{A}Y\) 表示操作 \(X\) 在顺序 A 中排在操作 \(Y\) 之前。

候选执行中的某些顺序必须与其他顺序一致。当且仅当不存在两个操作 \(X\) 和 \(Y\) 使得 \(X >_{A}Y\) 且 \(Y >_{B}X\) 时，顺序 A 与顺序 B 一致。

## 3.6 程序顺序

在任何单个执行单元 a 内部，存在一个事务的总程序顺序 \((\overline{po}_a)\)，该顺序与执行模型规定的程序控制流一致。

## 3.7 一致性顺序

对于任何段中的任何给定字节位置 L，在 HSA 无竞争程序中存在所有加载和存储的明显的、总的一致性顺序 \((\overline{coh}_L)\)。

我们也可以指一个单一的一致性顺序 \((\overline{coh})\)，定义为所有字节位置 \(\mathbb{L}\) 的所有按字节位置一致性顺序的简单并集 \((\overline{coh} = \bigcup_{\forall L \in \mathbb{L}} \overline{coh}_L)\)。

一致性顺序必须与程序顺序 \((\overline{po})\) 一致。

**[注释]** 此顺序包括普通操作和原子操作，并且在所有作用域实例中都是相同的。

**[注释]** 加载的每个字节将观察到由一致性顺序中最近的对同一字节位置的存储所产生的值。

## 3.8 全局依赖顺序

单个执行单元 a 的执行模型必须定义**本地依赖顺序** \((\overline{lda}_a)\) 的概念，该顺序定义了依赖操作。

**[注释]** 非正式地，如果操作 X 产生了一个结果，该结果被操作 Y 用于地址或数据计算，或者如果 X 影响了导致 Y 的控制流，则称操作 Y **依赖**于来自同一单个执行单元的操作 X。

我们定义了一个**全局依赖顺序** \((\overline{gdo})\)，它捕获了系统中代理之间存在的依赖关系。当且仅当满足以下条件时，操作 X 在全局依赖顺序中排在操作 Y 之前（形式上，等价于所有 \(\overline{lda}_a\) 和所有 \(\overline{coh}_L\) 的传递非自反闭包）：

- 对于某个执行单元 a，\(X >_{\overline{lda}_a} Y\)。
- X 和 Y 是访问字节位置 L 的内存事务，并且 \(X >_{\overline{coh}_L} Y\)。
- 存在一个操作 A 使得 \(X >_{\overline{gdo}} A\) 且 \(A >_{\overline{gdo}} Y\)。

根据规则，\(\overline{gdo}\) 中不能存在循环。

**[注释]** 无环要求防止出现操作 Y 似乎完成得比影响 Y 结果的操作 X 更早的情况。

**[注释]** 无环要求也禁止了**凭空产生值**。注意，这并不妨碍在所有内存操作的明显顺序中出现循环（即，非顺序一致的执行是可能的）。参见 3.13.3 非顺序一致执行（第 67 页）。

## 3.9 作用域同步顺序

在 HSA 无竞争程序的执行中，获取操作排在先前任何指定（直接或通过包含间接）相同作用域的释放操作之后。形式上，对于作用域实例 S，我们称之为**作用域同步顺序** \((\overline{SSO}_S)\)。

### 3.9.1 SC 原子释放与 SC 原子获取的同步

当且仅当满足以下所有条件时，原子操作 X 在作用域实例 S 的作用域同步顺序中排在原子操作 Y 之前：

- X 是具有释放或获取-释放语义的原子存储或读-改-写操作。
- Y 是具有获取或获取-释放语义的原子加载或读-改-写操作。
- X 和 Y 引用相同的位置 L 并且大小相同（例如，64 位）。
- X 和 Y 都指定（直接或通过包含间接）了作用域实例 S。
- 在字节位置 L 的操作一致性顺序中，\(X >_{\overline{coh}_L} Y\)。

### 3.9.2 释放栅栏与 SC 原子获取的同步

当且仅当满足以下所有条件时，栅栏 X 在作用域实例 S 的作用域同步顺序中排在原子操作 Y 之前：

- X 是具有释放或获取-释放语义的栅栏。
- Y 是对位置 L 进行的具有获取或获取-释放语义的原子操作。
- X 和 Y 都指定（直接或通过包含间接）了作用域实例 S。
- 存在对位置 L 的存储或读-改-写操作 A，使得 \(X >_{\overline{po}} A\)。
- A 和 Y 大小相同（例如，64 位）。
- 在 A 和 Y 共有的所有字节位置的总一致性顺序 \(\overline{coh_L}\) 中，\(A >_{\overline{coh}_L} Y\)。

### 3.9.3 释放 SC 原子与获取栅栏的同步

当且仅当满足以下所有条件时，原子操作 X 在作用域实例 S 的作用域同步顺序中排在栅栏 Y 之前：

- X 是对位置 L 的具有释放或获取-释放语义的原子存储或读-改-写操作。
- Y 是具有获取或获取-释放语义的栅栏。
- X 和 Y 都指定（直接或通过包含间接）了作用域实例 S。
- 存在从位置 L 的加载或读-改-写操作 A，使得 \(A >_{\overline{po}} Y\)。
- X 和 A 大小相同（例如，64 位）。
- 在 X 和 A 共有的所有字节位置的总一致性顺序 \(\overline{coh_L}\) 中，\(X >_{\overline{coh}_L} A\)。

### 3.9.4 两个栅栏的同步

当且仅当满足以下所有条件时，栅栏 X 在作用域实例 S 的作用域同步顺序中排在栅栏 Y 之前：

- X 是具有释放或获取-释放语义的栅栏。
- Y 是具有获取或获取-释放语义的栅栏。
- X 和 Y 都指定（直接或通过包含间接）了作用域实例 S。
- 存在对位置 L 的存储或读-改-写操作 A，使得 \(X >_{\overline{po}} A\)。
- 存在从位置 L 的加载或读-改-写操作 B，使得 \(B >_{\overline{po}} Y\)。
- A 和 B 大小相同（例如，64 位）。
- 在 A 和 B 共有的所有字节位置的总一致性顺序 \(\overline{coh_L}\) 中，\(A >_{\overline{coh}_L} B\)。

### 3.9.5 注释

**[注释]** 注意，对于特定的作用域，可能存在多个作用域实例。例如，来自两个不同工作组的工作项使用工作组作用域访问全局段，将看到工作组作用域实例的两个不同实例，并且可能看到两个不同的作用域同步顺序。

**[注释]** 在一次执行中，栅栏（它们没有关联的值）的完成顺序只能通过观察栅栏周围的加载和存储的顺序来确定。给定释放栅栏 R 和获取栅栏 A，R 和 A 只有在一个加载和存储观察到了顺序，使得 \(R >_{\overline{po}} S >_{\overline{coh}} L >_{\overline{po}} A\) 时，才能保证它们是同步的。正如在 3.12 无竞争程序的语义（第 55 页）中将显而易见的那样，存储和加载必须是原子的，以避免引入竞争，并且存储和加载很可能是宽松的（否则栅栏将是多余的）。

**[注释]** 如果还存在一个普通存储 S' 和普通加载 L'，使得 \(S^{\prime} >_{\overline{po}} R\) 且 \(A >_{\overline{po}} L^{\prime}\)，那么 S' 必须对 L' 可见（形式上，S' "HSA-发生在先" L'，参见 3.11 HSA-发生在先顺序（下方））。实际上，这意味着大多数实现不会将宽松存储与先前的释放栅栏重新排序，也不会将宽松加载与后续的获取栅栏重新排序，因为这样的重新排序可能导致 L' 在 S' 之前完成。

**[注释]** 注意，因为宽松原子可以在顺序一致的原子和栅栏之间创建一个作用域同步顺序，大多数实现会阻止宽松原子与顺序一致的原子在任何方向上重新排序。

## 3.10 顺序一致同步顺序

在候选执行中，在单个作用域实例中，所有具有释放、获取或获取-释放语义的同步操作存在一个总的明显顺序。此顺序称为**顺序一致同步顺序** \((\overline{sc}_S)\)。

给定同步操作 X 和 Y，如果 \(X >_{\overline{po}} Y\) 且 X 和 Y 指定了相同的作用域实例 S（直接或通过包含性间接），则 \(X >_{\overline{sc}_S} Y\)。

在 HSA 无竞争程序中，\(\overline{sc}_S\) 必须与以下各项一致：
- 程序顺序 \(\overline{po}\)
- 每个字节位置 L 的一致性顺序 \(\overline{coh}_L\)
- 所有其他顺序一致同步顺序 \(\overline{sc}_{S^{\prime}}\)，其中 \(S^{\prime}\neq S\)

## 3.11 HSA-发生在先顺序

我们定义了**HSA-发生在先** (hhb) 顺序，它捕获了同步内存操作之间的因果关系。

给定两个内存操作 X 和 Y，当且仅当以下条件之一成立时，\(X >_{hhb} Y\)：
- 存在操作 A 和 B，使得对于某个作用域实例 S，\(A >_{\overline{SSO}_S} B\)，并且
    - \(X >_{\overline{po}} A\) 或 X 就是 A 本身，并且
    - \(B >_{\overline{po}} Y\) 或 Y 就是 B 本身，或者
- 存在另一个操作 Z 使得 \(X >_{hhb} Z\) 且 \(Z >_{hhb} Y\)。

形式上，HSA-发生在先是作用域同步顺序和程序顺序的非自反传递闭包 \((\overline{hhb} = (\overline{po}\cup \overline{SSO})^{+})\)。

异构-发生在先与以下各项一致：
- 每个字节位置 L 的一致性顺序 \(\overline{coh}_L\)，以及
- 所有作用域实例 S 的所有顺序一致同步顺序 \(\overline{SC}_S\)。

在发生在先中不能存在循环。

**[注释]** 如果释放操作 X 在作用域实例 S 的作用域同步顺序中后跟获取操作 Y，则称这两个操作 X 和 Y 相对于作用域实例 S **同步**。

**[注释]** X 和 Y 之间的同步将 X 以及在程序顺序中先于 X 的所有事务，排在 Y 以及在程序顺序中后于 Y 的所有事务之前，形成一个单一的、全局可见的明显顺序，称为 HSA-发生在先。

**[注释]** 同步是传递的，因此两个执行单元 A 和 C 可以通过各自与执行单元 B 同步来间接同步。当涉及不同作用域时，只要每对释放/获取分别使用匹配的作用域，传递性也成立。例如，如果 A 通过工作组作用域与 B 同步，而 B 通过设备作用域与 C 同步，则 A 和 C 被认为是同步的。

## 3.12 无竞争程序的语义

给定一个程序 P，候选执行是所有那些存在以下顺序和规则且如后续章节所述保持一致的**可能执行**。

### 3.12.1 有效候选执行的定义

- **程序顺序** \((\overline{po}_a)\)：单个执行单元 a 的操作存在一个总顺序，该顺序尊重执行模型的语义。参见 3.6 程序顺序（第 51 页）。
- **顺序一致同步顺序** \((\overline{sc}_S)\)：在作用域实例 S 内的所有同步操作存在一个总的明显顺序，该顺序与执行中的每个程序顺序一致。参见 3.10 顺序一致同步顺序（第 54 页）。
- **一致性顺序** \((\overline{coh}_L)\)：对单个字节位置 L 的所有内存事务存在一个总的明显顺序，该顺序与候选执行中的每个顺序一致同步顺序和每个程序顺序一致。参见 3.9 作用域同步顺序（第 52 页）。
- **作用域同步顺序** \((\overline{SSO}_S)\)：对于每个作用域实例 S，存在一个特殊操作的明显顺序，该顺序定义了释放操作如何与获取操作同步。参见 3.9 作用域同步顺序（第 52 页）。
- **HSA-发生在先** (hhb)：存在一个发生在先关系，它捕获了同步的执行单元之间内存操作的因果关系。HSA-发生在先是所有程序顺序和所有作用域同步顺序的并集的非自反传递闭包：

    \[ \left(\left(\bigcup_{a\in A}\overline{po}_a\right)\cup \left(\bigcup_{S\in \S}\overline{SSO}_S\right)\right)^{+} \]

    其中 A 是所有执行单元的集合，\(\S\) 是候选执行中所有作用域实例的集合。

    hhb 顺序必须与候选执行中的每个一致性顺序一致。

    hhb 顺序必须与 \(\S\) 中所有 S 的所有顺序一致同步顺序 \(\overline{sc}_S\) 一致。

    在 hhb 中不能存在循环。

- **加载的值**：在一个明确定义的执行中，从字节位置 L 的加载（普通或原子）将始终观察到字节位置 L 的一致性顺序中最近的存储。
- **前进保证**：所有存储将在合理的有限时间内出现在一致性顺序中。

给定候选执行集合，我们可以通过检查竞争来确定程序 P 的实际执行是明确定义的还是未定义的。

### 3.12.2 冲突定义

- **普通冲突**：两个操作 X 和 Y 冲突，当且仅当它们访问一个或多个公共字节位置，其中至少一个是写入，并且至少一个是普通数据操作。
- **特殊冲突**：两个特殊操作 X 和 Y 冲突，当且仅当 X 和 Y 访问相同的字节位置，并且：
    - X 和 Y 大小不同（例如，32 位 vs. 64 位），或
    - 至少一个是写入（或读-改-写），并且 \(\sim \text{Match}(SI(X), SI(Y))\)。

### 3.12.3 未定义操作

- **未定义操作**：任何操作 X 是未定义的，如果：
    - 存在一个未定义操作 Y 使得 \(Y >_{\overline{gdo}} X\)。注意，任何依赖于未定义值的操作本身是未定义的。
- **未定义存储**：存储 S（原子或普通）是未定义的，如果：
    - 上述定义的未定义操作的一般规则适用，或
    - S 和另一个存储 S' 发生普通或特殊冲突，并且 S 和 S' 在 hhb 中是无序的。
- **未定义加载**：加载 L（普通或原子）是未定义的，如果：
    - 上述定义的未定义操作的一般规则适用，或
    - 相对于 L 在一致性顺序中最近的存储 S 是未定义的（如果加载读取了由未定义存储写入的值，则该加载是未定义的），或
    - L 和存储 S 发生普通或特殊冲突，并且 L 和 S 在 hhb 中是无序的。注意，如果加载是普通的并与普通或原子存储发生竞争，或者如果加载是原子的并与普通存储发生竞争，或者存在作用域不匹配，则该加载是未定义的。

### 3.12.4 竞争

- **HSA-竞争**：当且仅当普通或原子加载是未定义的时候，发生 HSA-竞争。
- **HSA-无竞争执行**：当且仅当不存在 HSA-竞争时，执行是 HSA-无竞争的。
- **HSA-无竞争程序**：当且仅当程序的所有有效候选执行都是 HSA-无竞争的时候，该程序是 HSA-无竞争的。

### 3.12.5 程序语义

- **明确定义和未定义的执行**：一次执行要么是明确定义的，要么是未定义的。如果存在任何操作 X 满足以下任一条件，则该执行是未定义的：
    - X 是加载或存储操作，且 X 的地址是未定义操作的结果，或
    - X 是图像加载或存储操作，且图像操作 X 的任何坐标是未定义操作的结果，或
    - X 是条件操作，且 X 的条件是未定义操作的结果。
- **明确定义和未定义的 HSA 程序**：当且仅当程序的所有有效候选执行都是明确定义的执行时，该 HSA 程序是明确定义的；否则，它是未定义的。
- **明确定义的 HSA 程序的语义**：明确定义的 HSA 程序的实际执行将是有效候选执行之一。
- **顺序一致性**：不使用宽松原子的 HSA-无竞争程序的执行将看起来是顺序一致的。加载的值是由在 hhb 中紧接在该加载之前的最近的对同一位置的存储所产生的值。

### 3.12.6 推论

- 在一个明确定义的程序中，仅包含已定义操作且不包含宽松原子的执行子集将看起来是顺序一致的。
- 在一个 HSA-无竞争程序中，其中每个位置被仅由普通操作或仅由原子操作独占访问：
    - 普通加载观察到的值将是 HSA-发生在先中对该位置的唯一最近普通存储。
    - 原子加载观察到的值将来自一个存储 X，该存储是在 \(\overline{coh}_L\) 顺序中先于该加载的最近存储。存储 X 要么与该加载在 HSA-发生在先中有序，要么无序。如果存储 X 与该加载在 HSA-发生在先中有序，则存储 X 必须在 HSA-发生在先中先于该加载，并且不会有其他存储 X' 在 HSA-发生在先中出现在 X 和该加载之间（即，X 将是 HSA-发生在先偏序中多个最近存储之一）。
- 在一个 HSA-无竞争程序中，其中至少一个位置被普通和原子操作同时访问：
    - 普通加载观察到的值将来自 HSA-发生在先中对该位置的最近存储之一。可能存在多个这样的存储。
    - 原子加载观察到的值与上述相同。

#### 3.13.1.1 同步操作按定义是顺序一致的

考虑同一工作组中的四个执行单元 A、B、C 和 D，以及两个内存位置 X 和 Y：

```pseudo

atomic_store(Y, s1, rlx, system);

B:

X = 1;

B:
s1 = X;
```



hsail

```
A:
st_global_s32 1, [&X];

B:
ld_global_s32 $s1, [&X];
```



A 对 X 的写入和 B 对 X 的读取形成了一个普通冲突。没有同步，这意味着来自 A 和 B 的操作在 HSA-发生在先 (hhb) 中是无序的。因此，此示例包含一个竞争。

#### 3.13.4.2 作用域不足

考虑不同工作组中的两个执行单元 A 和 B，以及两个内存位置 X 和 Y：

pseudo

```
global int X = 0;
global int Y = 0;

A:
X = 1;
atomic_store(Y, 1, screl, wg);

B:
while (atomic_load(Y, scacq, wg) != 1) { }
s2 = X; // races with X = 1
```



hsail

```
global_s32 &X=0;
global_s32 &Y=0;

A:
st_global_s32 1, [&X];
atomicnoret_st_global_screl_wg_s32 [&Y], 1;

B:
@b_start:
atomic_ld_global_scacq_wg_s32 $s1, [&Y];
cmp_ne_b1_s32 $c1, $s1, 1;
cbr_b1 $c1, @b_done;
ld_global_s32 $s2, [&X];
```



此示例中有两个竞争。首先，B 的原子加载与 A 的原子存储发生竞争，因为工作组作用域不具有包含性（B 在另一个工作组，工作组作用域不匹配）。其次，由于用于同步的作用域同样不匹配，对普通变量 X 的写入和读取形成竞争。

# 附录 A. 限制

本附录列出了 HSA 实现必须支持的最大或最小值：

- **波前大小**：实现中每个内核代理的每种调用约定都必须具有一个波前大小，该大小是 2 的幂，范围在 1 到 256 之间（含）。
- **工作组维度和大小**：每个工作组维度限制为 216−1216−1。工作组大小是三个单独工作组维度的乘积，限制为 232−1232−1。实现中的每个内核代理都必须支持至少 256 的单个工作组维度和工作组大小。不超过所支持工作组大小的、受支持的单个工作组维度的每种排列都必须被支持。
- **网格维度和大小**：每个网格维度限制为 232−1232−1。网格大小是三个单独网格维度的乘积，限制为 264−1264−1。实现中的每个内核代理都必须支持至少与相应受支持工作组大小相同的单个网格维度和网格大小。不超过所支持网格大小的、受支持的单个网格维度的每种排列都必须被支持。
- **组段内存大小**：实现中的每个内核代理必须为每个计算单元支持至少 32K 字节的组段内存，用于组段变量。如果实现将组内存用于其他 HSAIL 功能的实现（例如细粒度屏障和异常检测），则此数量可能会减少。
- **私有段内存大小**：实现中的每个内核代理必须为每个工作项支持至少以下较小值的字节数：
  - 256 字节
  - 64K 除以由关联的内核派发数据包指定的工作组大小
    由于要求实现中的每个内核代理支持最多 256 个工作项的工作组，并且必须一次至少执行一个工作组，这意味着内核代理必须能够支持至少 64K 字节的私有段内存。
- **kernarg 段内存大小**：每个实现必须为每个派发支持至少 1K 字节的 kernarg 段内存。
- **图像数据类型支持**：每个支持图像的代理必须支持具有以下每代理限制的图像：
  - 1D 图像：必须支持 1D、1DA 图像尺寸至少为宽度 16384 个图像元素。
  - 2D 图像：必须支持 2D、2DA、2DDEPTH、2DADEPTH 图像尺寸至少为（宽度，高度）= (16384 x 16384) 个图像元素。
  - 3D 图像：必须支持 3D 图像尺寸至少为（宽度，高度，深度）= (2048 x 2048 x 2048) 个图像元素。
  - 图像数组：必须支持 1DA、2DA、2DADEPTH 图像数组，数组大小至少为 2048 个图像层。
  - 1DB 图像：必须支持 1DB 图像尺寸至少为宽度 65536 个图像元素。
  - 只读图像句柄：必须支持在任何时候创建至少 128 个只读图像句柄。
  - 只写和读写图像句柄：必须支持在任何时候创建总共至少 64 个只写和读写图像句柄的组合。
  - 采样器句柄：必须支持在任何时候创建至少 16 个采样器句柄。

可以使用 HSA 运行时查询来确定特定代理实际支持的限制。请注意，图像或采样器的创建可能由于资源不足而失败。例如，代理可能无法以其支持的图像大小创建图像，因为内存不足。

# 附录 B. 术语表

**获取同步操作**
指定获取内存排序的内存操作。

**代理**
参与 HSA 内存模型的硬件或软件组件。代理可以提交 AQL 数据包以供执行。代理也可以是（但不要求是）内核代理。系统中可能包含既不是内核代理也不是主机 CPU 的代理。

**应用全局内存**
将在所有代理和主机 CPU 之间共享以供 HSA 处理使用的内存。这对应于全局段。

**架构化队列语言 (AQL)**
AQL 数据包是 HSA 标准的数据包格式。AQL 内核派发数据包用于在内核代理上派发内核，并指定启动维度、内核代码句柄、内核参数、完成检测等。其他 AQL 数据包控制内核代理的各个方面，例如何时执行 AQL 数据包以及使内存操作的结果可见。AQL 数据包在用户模式队列中入队。参见 2.9 需求：架构化队列语言 (AQL)（第 24 页）。

**arg 段**
用于将参数传入和传出函数的内存段。

**计算单元**
能够执行 HSAIL 指令集的虚拟硬件单元。一个工作组的工作项在同一个计算单元上执行。内核代理由一个或多个计算单元组成。

**全局段**
内存对所有代理中的所有执行单元可见的内存段。

**网格**
一个包含工作组的多维矩形结构。当程序启动内核时会形成一个网格。

**组段**
内存对单个工作组可见的内存段。

**主机 CPU**
一种代理，也支持原生 CPU 指令集，并运行主机操作系统和 HSA 运行时。作为代理，主机 CPU 可以使用内存操作构建 AQL 数据包并将其入队，从而向内核代理派发命令。在某些系统中，主机 CPU 也可以充当内核代理（具有适当的 HSAIL 最终化器和 AQL 机制）。

**HSA 应用程序**
用主机 CPU 指令集编写的程序。除了主机 CPU 代码外，它还可能包含零个或多个 HSAIL 程序。

**HSA 实现**
一个或多个能够执行 HSA 运行时的主机 CPU 代理、一个或多个能够执行 HSAIL 程序的内核代理，以及零个或多个参与 HSA 内存模型的其他代理的组合。

**HSA 内存管理单元 (HSA MMU)**
内核代理和其他代理使用的内存管理单元，设计用于支持具有与主机 CPU MMU 提供的相同属性的页粒度虚拟内存访问。

**HSA 内存节点 (HMN)**
一个节点，它划分了一组对连接到该节点内存控制器的内存资源集具有 "本地" 访问权限的系统组件（主机 CPU 和内核代理），并具有适当的 HSA 兼容访问属性。

**HSAIL**
异构系统架构中间语言。一个虚拟机及一种语言。HSA 虚拟机的指令集，保留了虚拟机抽象，并允许廉价地转换为机器码。

**无效地址**
应用全局内存中的一个位置，内核代理或其他代理对该位置的访问违反了由系统页表属性设置所建立的系统软件策略。如果内核代理访问了一个无效地址，系统软件应得到通知。参见 2.1 需求：共享虚拟内存（第 11 页）和 2.9.3 错误处理（第 27 页）。

**kernarg 段**
用于将参数传入内核的内存段。

**内核**
由内核代理以数据并行方式执行的代码段。内核用 HSAIL 编写，并由最终化器转换为机器码。

**内核代理**
支持 HSAIL 指令集并支持执行 AQL 内核派发数据包的代理。作为代理，内核代理可以使用内存操作构建 AQL 数据包并将其入队，从而向任何内核代理（包括其自身）派发命令。内核代理由一个或多个计算单元组成。

**内存类型**
由转换表定义的一组属性，至少涵盖以下属性：

- 可缓存性
- 数据一致性要求
- 端点排序要求（内存访问到达端点的顺序）
- 观察排序要求（对不同位置的内存访问的可观察顺序的限制）
- 多副本原子性要求
- 推测性访问的权限

**自然对齐**
大小为 n*n* 字节的内存操作的地址是 n*n* 的整数倍的对齐方式。例如，自然对齐的 8 字节存储只能指向地址 0、8、16、24、32、40 等。

**数据包 ID**
每个 AQL 数据包都有一个 64 位的数据包 ID，在其入队的用户模式队列中是唯一的。数据包 ID 被分配为用户模式队列中分配的逻辑数据包槽位的单调递增序列号。数据包 ID 和队列 ID 的组合对于进程是唯一的。

**数据包处理器**
数据包处理器紧密绑定到一个或多个代理，并提供处理在这些代理的用户模式队列中入队的 AQL 数据包的功能。数据包处理器的功能可以由与用户模式队列关联的（将执行内核派发数据包或代理派发数据包功能的）代理相同的代理执行，也可以由不同的代理执行。

**主内存类型**
用户进程默认使用的内存类型。代理对此类型的数据一致性（不包括对只读图像数据的访问）和可缓存性属性应有共同解释。

**私有段**
内存仅对单个工作项可见的内存段。用于读写内存。

**进程地址空间 ID (PASID)**
用于标识主机 CPU 或客户虚拟机内的应用程序地址空间的 ID。它在设备上用于隔离驻留在共享本地内存中的并发上下文。

**队列 ID**
进程中用户模式队列的标识符。每个队列 ID 在进程中都是唯一的。队列 ID 和数据包 ID 的组合对于进程是唯一的。

**readonly 段**
用于只读内存的内存段。

**释放同步操作**
指定释放内存排序的内存操作。

**段**
连续的可寻址内存块。段具有大小、可寻址性、访问速度、访问权限以及工作项之间的共享级别。也称为内存段。

**信号句柄**
一个不透明的句柄，指向一个信号，该信号可用于属于单个进程但可能在 HSA 系统中不同代理上执行的线程和工作项之间的通知。

**溢出段**
用于加载或存储寄存器溢出的内存段。

**执行单元**
通过一个处理元素的程序排序操作序列。执行单元可以是代理上的任何执行线程、一个工作项，或者是通过 HSA 兼容设备中的处理元素发送操作的任何方法。

**用户模式队列**
用户模式队列是由 HSA 运行时创建的内存数据结构，AQL 数据包可以在其上入队。数据包由与用户模式队列关联的数据包处理器处理。例如，与内核代理的数据包处理器关联的用户模式队列可用于在该内核代理上执行内核。参见 2.8 需求：用户模式队列（第 17 页）。

**波前**
在单个程序计数器上执行的一组工作项。

**工作组**
由内核派发形成的工作项网格的一个分区。它是计算单元中的一个执行实例。

**工作项**
由内核派发形成的网格的单个执行单元。
